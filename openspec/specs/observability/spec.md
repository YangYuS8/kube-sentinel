# observability 规范

## 目的

定义 Kube-Sentinel 极简 V1 的基础观察面，确保事件、审计和指标首先服务夜间值班理解与人工接管，而不是发布治理与周期性汇报。

## 需求

### 需求:基础观察面必须服务值班理解与人工接管
系统必须将基础观察面限定为帮助值班人员快速理解 incident、支持 Agent 生成五段式输出并完成人工接管，至少覆盖触发情况、最小动作结果、阻断原因、快照结果、最近相关事件和稳定关联键，禁止将发布治理趋势视为 V1 基础观察面的前置内容。

#### 场景: Agent 或值班人员查看单个 incident 的观察证据
- **当** Agent 或值班人员查看某个 incident 的运行证据
- **那么** 系统必须提供与该 incident 相关的基础事件、审计记录、关键指标和关联键

### 需求:控制台指标必须具备标准抓取入口

系统必须以标准 Prometheus 抓取方式暴露控制器基础指标，使 Grafana 能够通过 Prometheus 数据源消费 incident 触发、动作结果、阻断原因和快照结果等最小运行态指标，禁止仅依赖临时端口转发或人工拼接查询规则作为唯一接入方式。

#### 场景: 测试环境接入 Grafana

- **当** 测试环境启用 Grafana 面板查看 Kube-Sentinel 运行态指标
- **那么** 系统必须提供标准的指标抓取入口与最小接入配置，使 Prometheus 能稳定抓取 incident 相关的基础控制器指标

### 需求:指标视图必须覆盖最小运维分组

系统必须为 Grafana 等指标视图定义最小运维分组，至少覆盖总体触发与成功率、L1 动作结果、快照结果和阻断趋势，禁止将发布治理或复杂交付趋势作为 V1 最小仪表盘的必需内容。

#### 场景: 运维查看最小仪表盘

- **当** 运维打开 Kube-Sentinel 的最小运行态仪表盘
- **那么** 仪表盘必须能够按分组展示总体触发、L1 结果、快照结果与阻断趋势四类视图

### 需求:基础观察面必须支持 Agent 焦点判断
系统必须为 Agent v1 提供最小观察证据集合，用于支撑 `current focus` 和 `next steps` 的生成，至少覆盖动作结果、阻断原因、快照状态、最近相关事件和稳定关联键。

#### 场景: Agent 为 blocked incident 生成焦点判断
- **当** Agent 为一个 blocked incident 生成 `current focus`
- **那么** 系统必须能够提供足以区分安全阻断、启动失败、配置依赖问题或证据不足的最小观察证据

### 需求:Telegram incident card 必须可追踪到观测证据
系统必须使 Telegram incident card 能够追踪回对应的对象、事件、审计记录和趋势视图，并要求发送层在消息中保留这些继续排查入口，禁止让通知成为无法继续追查的孤立消息。

#### 场景: 值班人员从 Telegram 消息继续排查
- **当** 值班人员从 Telegram 收到某个 incident 的通知卡片
- **那么** 该卡片必须包含足以定位 `HealingRequest`、Grafana 趋势视图和精确排查入口的稳定关联信息

### 需求:Telegram 发送结果必须可观察
系统必须为 Telegram 通知发送结果提供最小可观察语义，至少包括发送成功/失败/抑制、关联键和目标 incident 身份，禁止让通知成为无法追踪的黑箱副作用。

#### 场景: 查询 Telegram 发送结果
- **当** 运维排查某个 incident 的 Telegram 通知是否成功送达
- **那么** 系统必须能够提供与该发送结果相关的最小可观察记录，并区分发送与抑制

### 需求:通知抑制结果必须可观察
系统必须为 Telegram 通知抑制和去重结果提供最小可观察语义，至少包括“已发送”“已抑制”“已失败”三类结果，禁止让降噪逻辑成为不可见的黑箱。

#### 场景: 查询某个 incident 的通知是否被抑制
- **当** 运维排查某个 incident 为什么没有再次发送 Telegram
- **那么** 系统必须能够说明该消息是已发送、已抑制还是发送失败
