## 需求
### 需求:统一发布就绪摘要
系统必须在每次发布评估后输出统一发布就绪摘要，至少包含 `allow/degrade/block` 结论、主要原因、风险等级与下一步动作建议。

#### 场景: 发布前读取摘要
- **当** 值班人员在发布窗口执行放量前检查
- **那么** 系统必须返回可直接决策的发布就绪摘要

### 需求:值班动作模板映射
系统必须将门禁结论映射到固定值班动作模板，至少包含告警级别、runbook、审批触发点与回滚动作建议。

#### 场景: 结论映射模板动作
- **当** 发布门禁输出 `degrade` 或 `block`
- **那么** 系统必须输出对应模板动作并可被自动化流程解析

### 需求:演练结果自动汇总
系统必须将闭环演练结果汇总为可审计报告，至少包含通过项、失败项、阻断原因和后续观察计划。

#### 场景: 演练结束生成汇总
- **当** 主线演练脚本执行完成
- **那么** 系统必须生成可检索的汇总报告并关联发布窗口

### 需求:人工覆盖可追踪
系统必须允许在特殊情况下人工覆盖模板动作，但必须记录覆盖原因、审批人和恢复条件。

#### 场景: 人工覆盖模板动作
- **当** 值班人员执行人工覆盖
- **那么** 系统必须记录覆盖证据并纳入复盘输出

### 需求:发布判定与值班模板一致性

系统必须保证发布就绪判定结果与值班动作模板映射保持一致，并可被自动化流程校验。

#### 场景: 判定与模板不一致

- **当** 发布判定结论与值班模板等级或 runbook 不一致
- **那么** 系统必须判定失败并输出不一致项

### 需求:人工覆盖审计链路完整

系统必须在人工覆盖发生时记录覆盖人、覆盖前后判定、覆盖原因和时间戳，并与发布证据关联。

#### 场景: 人工覆盖发生

- **当** 值班人员执行人工覆盖
- **那么** 系统必须输出完整审计字段并可通过关联键检索
