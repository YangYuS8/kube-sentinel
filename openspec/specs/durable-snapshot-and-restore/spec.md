## 需求
### 需求:持久快照记录
系统必须在每次自动写动作前创建持久快照记录，且快照必须包含工作负载标识、目标阶段、版本证据、创建时间、过期时间与幂等键。

#### 场景: 创建持久快照成功
- **当** 编排器进入 L1 或 L2 写动作前置检查
- **那么** 系统必须创建并持久化快照记录

### 需求:快照恢复执行
系统必须支持基于快照的恢复执行；恢复流程必须具备可重试幂等性，并输出成功或失败证据。

#### 场景: 基于快照恢复成功
- **当** 自动动作失败并触发恢复
- **那么** 系统必须根据快照完成恢复并记录恢复结果

#### 场景: 基于快照恢复失败
- **当** 自动动作失败且恢复执行失败
- **那么** 系统必须进入冻结并升级到 L3 人工介入

### 需求:快照生命周期治理
系统必须提供快照保留窗口与数量上限治理；超过上限时必须触发清理或阻断策略，禁止无界增长。

#### 场景: 快照达到上限
- **当** 单个工作负载快照数量超过配置上限
- **那么** 系统必须按策略清理旧快照或阻断新动作
