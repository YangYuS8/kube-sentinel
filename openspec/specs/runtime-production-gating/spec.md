## 需求
### 需求:统一生产门禁判定模型
系统必须提供统一的生产门禁判定模型，并输出 `allow`、`block`、`degrade` 三类互斥结果。

#### 场景: 单次判定输出唯一结果
- **当** 系统接收一次发布门禁评估请求
- **那么** 系统必须仅输出一个最终判定结果且附带原因码

### 需求:门禁优先级顺序
系统必须按固定优先级执行门禁判定：维护窗口与安全门禁优先于熔断，熔断优先于发布阈值门禁。

#### 场景: 多门禁同时命中
- **当** 维护窗口、熔断与发布阈值门禁同时命中
- **那么** 系统必须按优先级输出最高优先级的阻断结论

### 需求:越线自动降级策略
系统在判定结果为 `degrade` 或 `block` 时必须执行保守降级动作，至少包括只读评估、结构化告警与人工介入建议。

#### 场景: 发布阈值越线触发降级
- **当** 发布阈值门禁判定为越线
- **那么** 系统必须切换到保守模式并停止自动写操作

### 需求:门禁证据标准化输出
系统必须在每次门禁判定后输出可检索证据，至少包含原因码、阈值快照、样本窗口、关键指标值与下一步建议。

#### 场景: 判定证据可追溯
- **当** 运维查询一次历史门禁判定
- **那么** 系统必须返回与该判定关联的完整证据字段

### 需求:发布判定绑定 API 契约风险
生产门禁必须将 API 契约判定纳入发布决策；当契约风险等级为高或兼容性要求未满足时必须阻断放量。

#### 场景: 契约高风险阻断发布
- **当** 发布门禁检测到 version-bump-required 但发布计划未包含版本切换窗口
- **那么** 系统必须输出 block 并禁止继续放量

### 需求:生产门禁证据补齐状态语义
生产门禁判定证据必须包含状态语义最小集合（阶段、最近动作、失败原因、下一步建议）以支持值班诊断。

#### 场景: 证据缺失下一步建议
- **当** 门禁判定结果生成但缺少下一步建议字段
- **那么** 系统必须将该判定视为不完整并阻断发布
