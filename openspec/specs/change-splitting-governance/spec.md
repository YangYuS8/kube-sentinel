# change-splitting-governance 规范

## 目的
待定 - 由归档变更 increment-splitting-governance 创建。归档后请更新目的。
## 需求
### 需求:变更拆分触发阈值

系统必须在提案阶段对变更规模执行拆分判定，当 capability 数量大于等于 3 或增量需求条目总数大于 10 时，必须要求拆分为多个变更。

#### 场景: 命中 capability 数量阈值

- **当** 提案声明修改或新增的 capability 数量达到 3
- **那么** 系统必须给出“需要拆分”的判定并提示拆分模板

#### 场景: 命中增量条目阈值

- **当** 单次变更增量需求条目总数超过 10
- **那么** 系统必须给出“需要拆分”的判定并阻止直接进入归档

### 需求:高风险混合变更强制拆分

系统必须禁止将“发布阻断主链能力”与“运营增强能力”合并在同一变更中，除非提供例外审批证据。

#### 场景: 风险域混合且无审批

- **当** 变更同时包含阻断主链与运营增强能力，且未提供例外审批
- **那么** 系统必须判定不通过并要求拆分

#### 场景: 风险域混合但有审批

- **当** 变更同时包含阻断主链与运营增强能力，且提供了例外审批与风险说明
- **那么** 系统必须允许继续并保留审计记录

### 需求:归档前拆分自检清单

系统必须在归档前输出拆分自检结果，至少包含范围复杂度、风险耦合度、评审可读性与回滚影响面四项。

#### 场景: 自检项不完整

- **当** 归档前缺失任一必填自检项
- **那么** 系统必须阻止归档并提示补齐自检信息

#### 场景: 自检通过

- **当** 四项自检全部完成且无强制拆分冲突
- **那么** 系统必须允许进入归档流程
