# v1-release-execution 规范

## ADDED Requirements

### 需求:首发版本边界必须固定

系统必须将首个可交付版本的范围固定为 Deployment 自动处置闭环，禁止将 StatefulSet 自动写动作、额外发布控制平面或独立 UI 查询层作为首发完成前置条件。

#### 场景: 宣布首发范围

- **当** 团队定义或执行首个可交付版本的发布计划
- **那么** 系统必须明确说明首发仅覆盖 Deployment 自动闭环，并明确列出不属于首发阻断范围的能力

#### 场景: 首发范围被扩大

- **当** 首发发布计划试图将 StatefulSet 自动写动作或额外 UI 后端纳入首发阻断项
- **那么** 系统必须阻止将其视为首发必需能力，并要求另立变更或后续版本计划

### 需求:首发发布顺序必须统一

系统必须将首发发布流程固定为“预生产验证通过 -> 发布 RC -> 执行 pilot/cutover -> 发布稳定版”，禁止跳过任一阶段直接发布稳定版。

#### 场景: 按顺序推进首发发布

- **当** 首发版本进入正式发布流程
- **那么** 系统必须要求先完成预生产全链路验证与 go-live 判定，再发布 RC，并在 pilot/cutover 通过后才允许稳定版发布

#### 场景: 试图跳过 RC 或 pilot

- **当** 操作者试图在未完成预生产验证、RC 发布或 pilot/cutover 的情况下直接发布稳定版
- **那么** 系统必须将该流程视为无效首发发布路径并要求回到统一顺序执行

### 需求:首发证据包必须最小完备

系统必须为首发版本生成最小完备的发布证据包，至少包含 go-live decision pack、pilot/cutover decision pack、预生产演练输出、版本标签记录和回滚说明。

#### 场景: 首发证据包完整

- **当** 团队完成一次首发发布预演或正式发布
- **那么** 系统必须能够提供完整证据包并支持回顾该版本的放行依据与回滚路径

#### 场景: 首发证据包缺失

- **当** 首发发布记录缺少任一最小证据项
- **那么** 系统必须禁止将该版本声明为首个可交付版本，并要求补齐发布证据

### 需求:首发回滚基线必须与发布记录绑定

系统必须为首发版本显式记录回滚基线，包括回滚目标版本、禁用自动化的最小步骤和证据关联键，禁止以临时口头说明替代正式回滚记录。

#### 场景: 首发发布成功后记录回滚基线

- **当** 首发版本进入 RC 或稳定版发布阶段
- **那么** 系统必须同步记录对应的回滚基线与回滚说明，并能通过关联键定位对应审计与决策包

#### 场景: 首发发布时缺少回滚基线

- **当** 发布流程未能给出明确的回滚目标和回滚步骤
- **那么** 系统必须阻断首发发布并要求先补齐回滚记录
