## ADDED Requirements

### 需求:V1 上线闸门统一判定

系统必须在发布前执行统一 go-live 判定，至少覆盖质量门禁、运行稳定性、演练与回滚、审批与冻结、审计完整性五类检查，任一类失败必须输出 block。

#### 场景: 任一闸门失败

- **当** go-live 判定中任一检查类返回失败
- **那么** 系统必须输出 block 结论并提供失败分类与修复建议

#### 场景: 全部闸门通过

- **当** 五类检查全部通过
- **那么** 系统必须输出 allow 结论并允许进入生产灰度

### 需求:发布决策包最小字段

系统必须生成结构化 release decision pack，至少包含最终结论、失败类别、回滚候选、演练结果摘要、审批状态、关联键与时间戳。

#### 场景: 生成决策包

- **当** 完成一次 go-live 判定
- **那么** 系统必须输出可解析的决策包并可用于复盘

#### 场景: 决策包字段缺失

- **当** 决策包缺失最小字段
- **那么** 系统必须阻断发布并提示补齐字段

### 需求:预生产到生产单向放量

系统必须要求“预生产验证通过”作为生产灰度放量前置条件，禁止跳过预生产验证直接进入生产放量。

#### 场景: 预生产未通过尝试放量

- **当** 预生产验证结果不为通过
- **那么** 系统必须阻断生产灰度并输出失败原因

#### 场景: 预生产通过后放量

- **当** 预生产验证通过且 go-live 判定为 allow
- **那么** 系统必须允许按灰度策略进入生产放量

### 需求:首发 go-live 必须绑定首发证据基线

系统必须将首发版本的 go-live 判定与首发证据基线绑定，至少校验预生产演练输出、release decision pack、目标版本标识和回滚说明已经齐备，禁止仅凭单次 CI 通过就宣称首发可放行。

#### 场景: 首发 go-live 证据齐备

- **当** 团队执行首发版本的 go-live 判定
- **那么** 系统必须同时验证质量门禁结果、预生产演练证据、目标版本标识和回滚说明，全部齐备后才允许输出 allow

#### 场景: 首发 go-live 缺少发布证据

- **当** go-live 判定缺少首发所需的任一证据项
- **那么** 系统必须输出 block，并要求先补齐证据后再继续首发发布
