## ADDED Requirements

### 需求:保守模式验收矩阵

系统必须提供覆盖高风险失败场景的保守模式验收矩阵，至少包含证据不足、输入缺失、预算阻断、熔断恢复边界与影子执行说明验证。

#### 场景: 高风险场景回归

- **当** 执行闭环验收演练
- **那么** 系统必须输出每个高风险场景的通过/失败结果与证据链完整性结论

### 需求:误动作与误阻断指标门槛

系统必须在验收报告中给出误放行率、误阻断率与重复执行率，并设置明确门槛用于发布阻断。

#### 场景: 指标超阈值阻断发布

- **当** 任一关键验收指标超出阈值
- **那么** 系统必须阻断发布并要求人工评审

## MODIFIED Requirements

### 需求:运行时闭环验收基线

系统必须提供可重复执行的闭环验收流程，并增加“参数配置化与事实输入化”强断言；在保守模式下必须验证 `PendingVerify`/`Suppressed` 路径行为。

#### 场景: 演练配置化校验

- **当** 运行主线演练脚本
- **那么** 系统必须验证门禁与幂等阈值来自请求配置且与输出状态一致

#### 场景: 观察期状态校验

- **当** 告警进入观察期并在窗口内恢复
- **那么** 系统必须记录 `Suppressed` 且无写操作发生

### 需求:双层熔断有效性验证

系统必须验证对象级与域级熔断均可阻断后续自动写操作，并验证阈值来源于声明式配置；命名空间预算阻断必须纳入同一验证流程。

#### 场景: 配置阈值触发熔断

- **当** 连续失败达到请求中配置的熔断阈值
- **那么** 系统必须阻断后续自动写操作并输出阈值证据

#### 场景: 命名空间预算触发阻断

- **当** 不健康应用占比达到命名空间预算阈值
- **那么** 系统必须进入只读阻断并输出预算证据
