# statefulset-healthy-revision-rollback 规范

## 目的

待定 - 由归档变更 statefulset-phase3-l2-rollback 创建。归档后请更新目的。

## 需求

### 需求:StatefulSet L2 健康版本候选筛选

系统必须在执行 StatefulSet L2 回滚前完成候选版本筛选，且候选必须同时满足稳定窗口与依赖完整性校验；该候选选择逻辑必须经过真实 Kubernetes ControllerRevision 历史验证，不能仅由 fake client 行为证明。

#### 场景: 候选筛选通过

- **当** StatefulSet 存在满足健康条件的历史版本
- **那么** 系统必须将该版本记录为 L2 回滚候选并进入回滚执行阶段

#### 场景: 候选筛选失败

- **当** StatefulSet 不存在满足健康条件的历史版本
- **那么** 系统必须禁止 L2 并降级到 L3 人工介入

#### 场景: 真实集群历史与当前假设不一致

- **当** 真实性验证发现 ControllerRevision 历史、候选排序或依赖可见性与当前实现假设不一致
- **那么** 系统必须修正候选筛选依据，并在无法证明安全时降级到 L3，而不是继续使用不成立的候选

### 需求:StatefulSet L2 回滚执行与恢复

系统必须在 L2 执行前创建快照；回滚失败时必须恢复快照并记录失败证据。该执行与恢复语义必须能够在真实集群中重复验证。

#### 场景: L2 回滚成功

- **当** 系统执行 StatefulSet L2 回滚成功
- **那么** 系统必须更新状态为已完成并记录目标健康版本

#### 场景: L2 回滚失败

- **当** 系统执行 StatefulSet L2 回滚失败
- **那么** 系统必须恢复快照、进入冻结并回退只读

#### 场景: 真实集群回滚失败后恢复成功

- **当** StatefulSet 在真实集群中的 L2 回滚失败，但快照恢复成功
- **那么** 系统必须保留失败与恢复联合证据，并阻断后续自动写操作直至人工确认

### 需求:StatefulSet L2 失败冻结与人工解锁

系统必须在 L2 失败后进入冻结窗口，冻结期间禁止任何自动写操作，直到人工解锁；该冻结与解锁语义必须在真实集群验证入口中可观测、可重复。

#### 场景: 冻结窗口阻断

- **当** StatefulSet 处于 L2 失败冻结窗口
- **那么** 系统必须阻断后续自动动作并输出剩余冻结时间

#### 场景: 真实性验证输出冻结证据

- **当** 真实性验证命中 StatefulSet L2 失败冻结路径
- **那么** 系统必须输出冻结状态、冻结截止时间和人工解锁要求，作为闭环验证证据的一部分
