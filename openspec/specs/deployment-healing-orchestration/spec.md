## 目的

定义 Deployment 与 StatefulSet 的统一编排入口与执行约束，保证多工作负载场景下的策略分流清晰、执行幂等且可追踪。

## 需求

### 需求:多工作负载统一编排入口

系统必须提供统一编排入口处理 Deployment 与 StatefulSet，并通过能力级别控制具体动作路径。

#### 场景: 按能力级别分流

- **当** 编排器接收到不同工作负载请求
- **那么** 系统必须根据能力级别分流到对应可写/只读路径

### 需求:编排链路幂等执行

系统必须保证多工作负载下同一告警重复触发不会导致重复副作用；在 StatefulSet L2 模式下，必须对同一窗口内重复回滚执行进行幂等阻断；对快照创建也必须使用同一幂等键避免重复持久化；在 Deployment 分层模式下，必须阻断同一窗口内重复阶段写动作。

#### 场景: StatefulSet L2 重复触发

- **当** StatefulSet 在同一幂等窗口重复触发 L2 回滚
- **那么** 系统必须拒绝重复回滚并输出幂等阻断证据

#### 场景: 快照重复创建请求

- **当** 同一工作负载在同一幂等窗口重复触发快照创建
- **那么** 系统必须复用或拒绝重复快照写入并记录原因码

#### 场景: Deployment 分层重复触发

- **当** Deployment 在同一幂等窗口重复触发同一阶段写动作
- **那么** 系统必须阻断重复执行并输出阶段阻断证据

### 需求:StatefulSet L1-L2-L3 阶段化编排

系统必须在统一编排状态机中支持 StatefulSet 的 L1→L2→L3 渐进式处置路径。

#### 场景: L1 失败后升级

- **当** StatefulSet L1 失败且满足升级条件
- **那么** 系统必须升级执行 L2 回滚判定

#### 场景: L2 无候选降级

- **当** StatefulSet L2 未找到健康候选或证据不足
- **那么** 系统必须降级到 L3 人工介入

### 需求:自动动作前持久快照校验

系统在 Deployment 与 StatefulSet 任一自动写操作执行前，必须先创建可持久快照；快照创建失败时必须禁止写操作并降级为只读评估。

#### 场景: 快照创建失败阻断写操作

- **当** 编排器在执行 L1 或 L2 前创建持久快照失败
- **那么** 系统必须阻断本次写操作并记录可诊断失败原因

### 需求:Deployment L1-L2-L3 阶段化编排

系统必须为 Deployment 提供显式 L1→L2→L3 渐进式处置状态机，且每次阶段迁移必须记录触发原因与证据。

#### 场景: L1 失败后升级 L2

- **当** Deployment L1 最小影响动作失败且门禁允许升级
- **那么** 系统必须进入 L2 回滚判定并记录升级依据

#### 场景: L2 证据不足降级 L3

- **当** Deployment L2 候选不足或依赖校验失败
- **那么** 系统必须降级到 L3 人工介入并输出建议

### 需求:Deployment L1 首发自动处置闭环

系统必须在首发版本中为 Deployment 提供单一 L1 自动动作闭环，并且该动作必须只在门禁通过、待验证窗口完成且写前快照创建成功后执行；待验证阶段的推进必须是可重复且冲突可控的，不得因为状态持久化竞争而破坏闭环。

#### 场景: Deployment L1 执行成功

- **当** Deployment 请求通过所有前置门禁、待验证窗口完成且写前快照创建成功
- **那么** 系统必须执行首发定义的唯一 L1 动作，并记录成功阶段与证据

#### 场景: 待验证窗口由控制器重试推进

- **当** Deployment 请求进入 PendingVerify 且尚未满足时间窗口或样本条件
- **那么** 系统必须在后续控制器重试中继续推进该请求，而不得依赖不受控的重复状态写回

### 需求:Deployment 首发版本禁止自动升级

系统必须在首发版本中禁止将 Deployment 从 L1 自动升级到更激进的自动动作；L1 失败或任一前置条件失败时必须输出阻断、降级或人工介入建议，而当请求后续重新满足条件并执行成功时，系统必须移除与旧阻断阶段冲突的状态语义。

#### 场景: Deployment L1 失败

- **当** 首发版本中的 Deployment L1 动作失败或任一前置条件失败
- **那么** 系统必须停止自动升级并输出可解释的失败原因与下一步建议

#### 场景: 后续成功覆盖旧阻断状态

- **当** 同一 Deployment 请求在之前失败或阻断后，经过条件修正再次执行并成功
- **那么** 系统必须以新的成功结果覆盖旧阻断语义，不得在最终状态中同时保留两套冲突结论

### 需求:Deployment 首发 L1 待验证阶段必须冲突可控

系统必须使用冲突可控的方式推进 Deployment 首发 L1 的 PendingVerify 阶段，禁止依赖高频状态自更新形成持续的对象版本冲突。

#### 场景: 待验证期间对象被并发修改

- **当** Deployment 首发 L1 请求处于 PendingVerify 阶段，且对象被人工 patch 或因控制器重入而并发修改
- **那么** 系统必须继续推进验证窗口或安全重试，而不得持续报出不可恢复的对象版本冲突

### 需求:Deployment 首发 L1 成功后必须输出收敛终态

系统必须在 Deployment 首发 L1 执行成功后输出收敛的 completed 终态，禁止保留仅适用于旧阻断阶段的失败原因、影子动作或门禁残留语义。

#### 场景: 门禁阻断后条件改善并执行成功

- **当** 同一 Deployment 请求先被门禁阻断，随后在新的条件下通过门禁并成功执行 L1
- **那么** 系统必须输出一致的 completed 状态，并只保留本次成功执行相关的证据与建议
