## 上下文

当前 Deployment 在实现层面已包含回滚与门禁逻辑，但缺少显式、可解释的 L1→L2→L3 阶段状态机定义，导致“策略升级原因、失败证据、降级建议”在运行时与规范层存在割裂。为对齐 `config.yaml` 的多级策略联动与可回滚优先原则，需要把 Deployment 路径升级为明确分层模型，并与已完成的 StatefulSet 分层语义保持一致。

## 目标 / 非目标

**目标：**
- 明确 Deployment 三阶段处置语义：L1 最小影响动作、L2 健康版本回滚、L3 人工介入。
- 统一 Deployment 分层路径的前置快照、失败恢复、幂等阻断与门禁联动。
- 提供可解释状态与证据：阶段决策、阻断原因、升级触发、下一步建议。
- 将 Deployment 分层能力纳入灰度发布、告警指标与闭环回归门禁。

**非目标：**
- 不扩展到 Deployment/StatefulSet 之外的新工作负载类型。
- 不引入新的外部存储或消息系统。
- 不在本变更中重构 Alertmanager ingestion 协议。

## 决策

### 决策 1：统一编排器内扩展 Deployment 分层状态机
- 方案：在现有 Orchestrator 中为 Deployment 路径引入显式 L1/L2/L3 状态转换与原因码。
- 原因：减少重复实现，保持 Deployment/StatefulSet 共享门禁与观测模型。
- 备选：新增 Deployment 专属编排器。未选原因：双实现易漂移，测试与维护成本更高。

### 决策 2：L1 失败才升级 L2，L2 证据不足则降级 L3
- 方案：L1 采用最小影响动作；只有 L1 失败且授权/门禁允许时进入 L2；候选不足或依赖缺失直接 L3。
- 原因：符合“渐进式处置”与“安全优先”。
- 备选：直接 L2 回滚。未选原因：动作更激进，误动作风险高。

### 决策 3：Deployment 分层沿用持久快照与幂等窗口
- 方案：L1/L2 写动作前必须创建快照；同窗口重复触发必须幂等阻断；回滚失败触发恢复并记录恢复结果。
- 原因：保证可回滚性与失败路径一致可验证。
- 备选：仅 L2 使用快照。未选原因：L1 失败无法统一恢复。

### 决策 4：发布门禁引入 Deployment 分层指标阈值
- 方案：新增 Deployment L1 成功率、L2 回滚成功率、L3 降级率与阻断率阈值，越线即降级保守模式。
- 原因：让生产启用可量化且可回退。
- 备选：仅人工观察日志。未选原因：不可持续且无法自动阻断。

## 风险 / 权衡

- [策略复杂度上升] 分层状态机条件增多 → 采用统一原因码与验收矩阵，保证可诊断。
- [误升级] L1 失败判定噪声导致频繁进入 L2 → 引入稳定窗口与幂等阻断。
- [告警风暴] 高频触发放大写操作 → 复用速率限制、熔断与门禁优先级。
- [灰度风险] 阈值设定不合理 → 先白名单灰度，并提供一键回退到只读策略。

## 迁移计划

1. 扩展 Deployment 分层策略与状态字段（API/Helm/CRD）。
2. 在 Orchestrator 增加 Deployment L1/L2/L3 显式状态迁移与证据写入。
3. 补齐分层指标、审计事件与最小告警集。
4. 更新闭环演练脚本与失败路径单测。
5. 灰度启用并以指标阈值作为发布阻断条件，越线则回退保守模式。

## Open Questions

- Deployment L1 最小影响动作默认选择“滚动重启”还是“单副本受控重建”更稳妥？
- L1→L2 升级是否需要额外的最小观测窗口以降低瞬时抖动影响？
- 是否需要为不同业务等级提供差异化 Deployment 分层阈值模板？
