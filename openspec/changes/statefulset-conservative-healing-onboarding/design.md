## 上下文

当前系统在 Deployment 上已经具备保守模式能力（观察期、证据门槛、预算阻断、影子执行），但 StatefulSet 仍未纳入同等治理路径。项目目标要求覆盖 Deployment/StatefulSet 两类核心工作负载，因此需要一个低风险接入方案：先实现 StatefulSet 只读闭环，再为后续受控自动动作打基础。

## 目标 / 非目标

**目标：**

- 建立 StatefulSet 的保守只读接入链路：接入、评估、证据、阻断、审计、事件、指标全链路可检索。
- 保证 StatefulSet 默认禁止自动写操作，命中时输出影子执行说明与人工介入建议。
- 让 Deployment 与 StatefulSet 共用核心门禁与观测语义，减少策略漂移。
- 提供可灰度、可回退、可验证的上线路径。

**非目标：**

- 不在本次开启 StatefulSet 的 L1/L2 自动写操作。
- 不在本次引入新的外部依赖或新的控制面组件。
- 不改动 Deployment 既有行为语义（仅做多工作负载兼容扩展）。

## 决策

1. **采用“两阶段策略”而非一步到位自动化**
   - 方案 A：直接支持 StatefulSet 自动动作。
   - 方案 B：先只读接入，再受控放开自动动作。
   - 选择 B。原因：符合“安全优先于自动化”，并可在真实运行数据上校准风险。

2. **在接入层显式分流 workload kind**
   - 方案 A：统一进入编排后再判定。
   - 方案 B：接入层先标注能力边界（read-only / writable）。
   - 选择 B。原因：更早阻断误动作，链路语义更清晰。

3. **复用现有保守门禁模型，增加工作负载维度输出**
   - 方案 A：为 StatefulSet 新建独立门禁逻辑。
   - 方案 B：复用现有门禁，输出增加 workloadKind 维度。
   - 选择 B。原因：降低复杂度，保证跨工作负载一致性。

4. **状态机新增只读意图表达，不新增写路径动作**
   - 方案 A：新增 StatefulSet 专属 phase。
   - 方案 B：沿用现有 phase，并通过 reason/shadowAction 表达只读阻断意图。
   - 选择 B。原因：最小化 CRD 变更，保持向后兼容。

## 风险 / 权衡

- [风险] StatefulSet 只读路径引入后，告警量可能上升 → [缓解] 增加分级告警与抑制窗口，默认 warning 级。
- [风险] 多工作负载共用门禁可能掩盖对象差异 → [缓解] 指标与事件携带 workloadKind 标签，支持分维度分析。
- [风险] 用户误以为已支持 StatefulSet 自动修复 → [缓解] 明确输出 shadowAction 与“manual-intervention-required”原因码。
- [风险] 后续 Phase 2 放开写操作时行为漂移 → [缓解] 在本次规格中提前定义迁移门槛与开关策略。

## Migration Plan

1. 在保持默认安全策略不变前提下，发布 StatefulSet 只读接入能力。
2. 灰度启用：先单命名空间，再扩大到低风险环境。
3. 观测验证：检查 StatefulSet 相关审计、事件、指标关联键完整性与误阻断率。
4. 回滚策略：关闭 StatefulSet 接入开关，回退到仅 Deployment 路径。
5. Phase 2 前置条件：稳定运行窗口内误放行率/误阻断率满足门槛，且人工审批流程可用。

## Open Questions

- StatefulSet 只读接入是否需要单独的全局 feature gate，还是复用现有策略开关。
- Phase 2 中允许的第一类自动动作应是受限重启还是受限回滚。
- 是否需要在 CRD 中显式声明“工作负载能力级别”（read-only / writable）以便外部系统读取。
