## 新增需求

### 需求:Deployment 首发 L1 待验证阶段必须冲突可控

系统必须使用冲突可控的方式推进 Deployment 首发 L1 的 PendingVerify 阶段，禁止依赖高频状态自更新形成持续的对象版本冲突。

#### 场景: 待验证期间对象被并发修改

- **当** Deployment 首发 L1 请求处于 PendingVerify 阶段，且对象被人工 patch 或因控制器重入而并发修改
- **那么** 系统必须继续推进验证窗口或安全重试，而不得持续报出不可恢复的对象版本冲突

### 需求:Deployment 首发 L1 成功后必须输出收敛终态

系统必须在 Deployment 首发 L1 执行成功后输出收敛的 completed 终态，禁止保留仅适用于旧阻断阶段的失败原因、影子动作或门禁残留语义。

#### 场景: 门禁阻断后条件改善并执行成功

- **当** 同一 Deployment 请求先被门禁阻断，随后在新的条件下通过门禁并成功执行 L1
- **那么** 系统必须输出一致的 completed 状态，并只保留本次成功执行相关的证据与建议

## 修改需求

### 需求:Deployment L1 首发自动处置闭环

系统必须在首发版本中为 Deployment 提供单一 L1 自动动作闭环，并且该动作必须只在门禁通过、待验证窗口完成且写前快照创建成功后执行；待验证阶段的推进必须是可重复且冲突可控的，不得因为状态持久化竞争而破坏闭环。

#### 场景: Deployment L1 执行成功

- **当** Deployment 请求通过所有前置门禁、待验证窗口完成且写前快照创建成功
- **那么** 系统必须执行首发定义的唯一 L1 动作，并记录成功阶段与证据

#### 场景: 待验证窗口由控制器重试推进

- **当** Deployment 请求进入 PendingVerify 且尚未满足时间窗口或样本条件
- **那么** 系统必须在后续控制器重试中继续推进该请求，而不得依赖不受控的重复状态写回

### 需求:Deployment 首发版本禁止自动升级

系统必须在首发版本中禁止将 Deployment 从 L1 自动升级到更激进的自动动作；L1 失败或任一前置条件失败时必须输出阻断、降级或人工介入建议，而当请求后续重新满足条件并执行成功时，系统必须移除与旧阻断阶段冲突的状态语义。

#### 场景: Deployment L1 失败

- **当** 首发版本中的 Deployment L1 动作失败或任一前置条件失败
- **那么** 系统必须停止自动升级并输出可解释的失败原因与下一步建议

#### 场景: 后续成功覆盖旧阻断状态

- **当** 同一 Deployment 请求在之前失败或阻断后，经过条件修正再次执行并成功
- **那么** 系统必须以新的成功结果覆盖旧阻断语义，不得在最终状态中同时保留两套冲突结论
