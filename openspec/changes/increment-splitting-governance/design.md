## 上下文

近期归档提示显示，单个变更跨越过多 capability 与需求条目时，评审复杂度明显上升，且难以在回滚与复盘阶段快速定位责任边界。当前流程缺少统一、可执行的“拆分触发阈值”和“例外审批路径”，导致是否拆分主要依赖个人经验。

## 目标 / 非目标

**目标：**

- 建立可执行的变更拆分规则，减少超大提案出现频率。
- 为高风险与高复杂度变更提供标准拆分模板，降低评审与归档成本。
- 在归档前增加自检清单，确保范围与风险透明。

**非目标：**

- 不修改控制器运行时逻辑、CRD 字段或任何业务执行链路。
- 不引入新的 CI 执行代码或额外部署组件。

## 决策

1. 采用“阈值触发 + 模板引导”的治理策略，而非仅保留软性建议。
   - 触发阈值：`capability >= 3`、`增量需求条目 > 10`、或“阻断主链 + 运营增强”混合变更。
2. 引入标准拆分模板：
   - 变更 A：发布阻断主链（pipeline/gate/dry-run/oncall）
   - 变更 B：运营增强链（observability/trend/nightly）
3. 保留紧急例外，但必须附带审批与风险说明，避免治理规则阻塞应急响应。
4. 将归档前自检定义为流程必经项，优先在 proposal/design 阶段暴露过宽范围。

## 风险 / 权衡

- [风险] 规则过严可能导致提案碎片化，增加管理开销  
  → 缓解：设置例外机制并要求最小审批证据。
- [风险] 团队短期内对新阈值理解不一致  
  → 缓解：给出明确样例与反例，并在首次执行时复盘。
- [权衡] 增加前置治理成本，换取后续评审与归档稳定性。

## 迁移计划

1. 在新 capability 中定义拆分治理需求与场景。
2. 在 `delivery-quality-gates` 增量规范中补充提案规模门禁语义。
3. 增加任务清单，要求在后续 apply 阶段实现流程检查与验收样例。
4. 若执行后发现阈值不合适，在下一次治理变更中调整参数。

## 开放问题

- 例外审批的最小证据应落在 proposal 还是 design 中，是否需要固定字段？
- “增量需求条目”是否按 capability 内条目计数，还是按全变更总数计数？
- 是否需要在 `openspec-cn status` 输出中直接暴露“建议拆分”原因分解？
