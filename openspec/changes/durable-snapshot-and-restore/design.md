## 上下文

当前自愈链路已具备 L1/L2/L3 渐进式处置、门禁与观测能力，但快照实现仍以进程内存为主，无法满足“完整备份、可恢复验证、可审计追踪”的生产要求。该短板直接影响回滚性校验的可信度：当控制器重启、实例漂移或长时间故障处置发生时，内存快照无法为恢复动作提供稳定依据。

本次设计在不改变“安全优先、可回滚优先、渐进式处置”原则下，将快照从占位能力升级为可持久、可治理、可验证的领域能力，并统一 Deployment/StatefulSet 的执行语义。

## 目标 / 非目标

**目标：**
- 在 L1/L2 任意写操作前，必须创建持久快照；创建失败时禁止写操作并降级。
- 为回滚失败路径提供确定性恢复流程，且恢复结果可审计、可检索、可指标化。
- 提供快照生命周期治理（保留窗口、清理、幂等键），控制存储与重复副作用风险。
- 将快照策略配置化，并纳入 CI 与演练门禁。

**非目标：**
- 不在本变更中引入跨集群快照复制。
- 不扩展到 Deployment/StatefulSet 之外的工作负载类型。
- 不替代现有发布系统或完整备份平台。

## 决策

### 决策 1：采用 K8s 原生持久对象承载快照元数据与恢复证据
- 方案：以命名空间内受控对象承载快照记录（包含 workload 标识、目标 revision、证据摘要、校验摘要、创建/过期时间、恢复状态）。
- 原因：与 Controller-runtime/ETCD 原生一致，便于 RBAC、审计、清理和故障后重建状态。
- 备选：外置 Redis/SQL 存储。未选原因：引入新依赖与运维复杂度，不符合当前范围边界。

### 决策 2：在 Orchestrator 中保持“快照→动作→恢复”统一事务语义（逻辑事务）
- 方案：对 L1/L2 路径统一执行前置快照，失败恢复通过同一快照键串联，所有阶段写入统一原因码。
- 原因：降低 Deployment/StatefulSet 分支差异，保持状态机可解释与测试可复用。
- 备选：在各适配器内各自处理快照。未选原因：职责分散、失败路径难以一致验证。

### 决策 3：引入快照幂等键与 TTL 清理
- 方案：快照键由 workload + phase + idempotency window 组成；到期后自动清理或后台清扫。
- 原因：满足“重复触发不重复副作用”，并避免快照无限积累。
- 备选：仅按时间戳写入。未选原因：重复告警下会产生冗余记录与恢复歧义。

### 决策 4：将快照策略参数配置化并贯通 API/Helm/CRD
- 方案：新增/扩展策略字段（如 enable、retentionMinutes、restoreTimeoutSeconds、maxSnapshotsPerWorkload），并同步 values/schema 与 CRD。
- 原因：满足去硬编码约束，支持灰度发布和风险控制。
- 备选：代码常量。未选原因：违反配置化规则，且不利于生产调优。

## 风险 / 权衡

- [存储膨胀] 快照记录数量增长过快 → 引入每对象上限、TTL 与清理指标，超过阈值阻断写操作。
- [恢复不一致] 目标对象在恢复期间再次变化 → 使用 resourceVersion/observedGeneration 校验并记录冲突原因码。
- [控制面负载] 高频告警导致额外 API 写入 → 复用幂等窗口、批量清理、门禁前置减少无效快照。
- [迁移风险] 旧路径与新路径并存阶段行为偏差 → 通过 feature gate 灰度，默认保守模式可一键回退。

## 迁移计划

1. 增加快照策略字段与默认值，生成并同步 CRD/Helm schema。
2. 引入持久快照实现并保留内存实现作为测试替身。
3. 将 Orchestrator L1/L2 路径切换到持久快照接口，补齐失败恢复与原因码。
4. 扩展指标、审计事件、告警规则与演练脚本断言。
5. 先在白名单命名空间灰度启用；若指标越线，回退到保守只读模式。
