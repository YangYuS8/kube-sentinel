## 上下文

当前实现已经满足 MVP 闭环验证，但在生产运行场景存在四类结构性问题：

1. 安全门禁与幂等窗口存在硬编码，配置无法完全驱动行为；
2. 门禁输入使用固定示例值，无法反映真实对象/集群状态；
3. Deployment 适配器未对接真实 Revision 与回滚动作；
4. 观测输出以内存结构为主，运维可采集能力不足。

本设计在不扩大功能边界的前提下，优先完成生产可用基线收敛：配置化、事实化、可采集化、可回滚化。

## 目标 / 非目标

**目标：**

- 去除门禁/幂等/熔断关键路径业务阈值硬编码，并统一由 CRD/配置驱动。
- 为编排链路提供真实运行态门禁输入（动作计数、影响 Pod、集群 Pod 基数）。
- 落地 Deployment Revision 查询与回滚执行，确保 L2 路径有真实数据来源。
- 将观测输出升级为可采集实现（Prometheus 指标与 K8s Event），并保持关联键贯穿。
- 补齐失败路径与边界测试，确保质量门禁可持续通过。

**非目标：**

- 不扩展 StatefulSet 自动写操作实现。
- 不引入跨集群联动治理或复杂策略 DSL。
- 不在本次变更内完成完整可观测平台建设。

## 决策

### 决策 1：参数优先级统一为“请求显式值 > 默认值 > 平台回退”

- 方案：所有门禁与幂等关键参数从 `HealingRequest.Spec` 读取，缺省时使用 `ApplyDefaults`，禁止在执行路径再次硬编码业务常量。
- 理由：满足配置可控与审计可解释，避免“文档配置与运行行为不一致”。
- 备选：继续在执行链路保留兜底常量；放弃，因会绕过 CRD/Helm 配置契约。

### 决策 2：门禁输入事实化

- 方案：在 Orchestrator 中注入运行态读取接口（计数与规模估算），用于维护窗口、限频、爆炸半径判定。
- 理由：将门禁由“静态演示”升级为“运行时判定”，满足安全边界要求。
- 备选：继续固定 `ActionsInWindow=0`、`AffectedPods=1`；放弃，因无法反映真实风险。

### 决策 3：DeploymentAdapter 生产化

- 方案：以 Client-go 读取 Deployment/ReplicaSet 历史 Revision 信息，提供最近健康候选输入，并执行回滚动作。
- 理由：让 L2 回滚链路具备真实执行语义，减少“逻辑通过但动作无效”。
- 备选：仅在 Orchestrator 内 mock Revision；放弃，因不满足生产可用目标。

### 决策 4：观测输出双轨制

- 方案：保留内存 sink 用于单测，同时增加 Prometheus 指标与 K8s Event 发射器的生产实现。
- 理由：兼顾可测试性和运维可采集性。
- 备选：全部替换为生产实现；放弃，因会增加测试复杂度并削弱单测稳定性。

## 风险 / 权衡

- [风险] 运行态输入采集失败导致误阻断或误放行 → [缓解] 采集失败时默认只读阻断并输出失败证据。
- [风险] Revision 选择逻辑升级导致回滚行为变化 → [缓解] 增加回归测试与灰度启用开关。
- [风险] 指标标签维度过多导致基数膨胀 → [缓解] 限制标签集合，仅保留资源/阶段/结果关键维度。
- [风险] 配置化后默认值变更影响旧行为 → [缓解] 显式记录默认值并补充兼容性说明。

## Migration Plan

1. 先引入参数读取与门禁输入接口，保持旧实现可回退。
2. 上线 DeploymentAdapter 生产实现并开启灰度，仅在低风险命名空间验证。
3. 启用 Prometheus + K8s Event 输出，核对与审计关联键一致性。
4. 通过主线演练与质量门禁后，移除执行路径硬编码。
5. 若出现异常，回退到“L1 + 人工介入 + 只读评估”模式。
