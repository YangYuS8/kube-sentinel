## 上下文

当前系统已经具备多层自愈、门禁阻断和闭环演练能力，但“通过门禁”仍主要是工程信号，尚未形成对生产稳定性负责的运营信号。发布决策缺少统一的 SLO 契约：何时可以继续放量、何时应降级、何时必须升级人工介入。为了让自愈系统从“可用”走向“可运营”，需要把 SLO、门禁结果与事故响应流程打通，并沉淀可复盘的证据链。

## 目标 / 非目标

**目标：**
- 定义运行时 SLO 治理契约（指标、预算、评估窗口、越线动作）。
- 将 `allow/block/degrade` 门禁结果与事故响应等级、通知语义、恢复条件建立一一映射。
- 在审计与闭环演练中固化“越线 -> 升级 -> 回滚/恢复”的证据路径。
- 提供可灰度启用的运行策略，避免一次性切换导致误阻断。

**非目标：**
- 不引入新的 L1/L2/L3 自愈动作算法。
- 不引入新的外部故障管理平台集成（如 PagerDuty 深度双向同步）。
- 不重写已有门禁实现，只在现有链路上补充治理语义与输出契约。

## 决策

1. 引入独立的 SLO 治理 capability，而不是把所有规则塞进质量门禁脚本。
- 原因：SLO 属于运行治理契约，不应与执行器耦合；可独立演进且便于审计。
- 备选：仅在 `quality-gate.sh` 中增加更多规则。
- 未选原因：边界混乱，无法表达“发布前通过但运行期越线”的状态转换。

2. 使用三态映射驱动事故响应等级。
- 方案：`allow` -> 正常观测；`degrade` -> 保守降级 + 值班告警；`block` -> 阻断发布 + 必要人工审批。
- 原因：与现有门禁语义一致，可减少新增概念。
- 备选：定义全新五级状态模型。
- 未选原因：学习成本高，短期收益有限。

3. 使用窗口化 SLO 预算而非单点阈值。
- 方案：在固定时间窗口内计算错误预算消耗与趋势，避免瞬时抖动触发误动作。
- 原因：自愈链路具有 burst 特性，单点阈值噪声高。
- 备选：每次事件即时判定是否越线。
- 未选原因：误报率高，影响发布节奏。

4. 将 SLO 断言纳入闭环演练矩阵。
- 方案：在演练脚本中验证越线分类、升级路径、恢复前置条件。
- 原因：保证策略不仅“定义正确”，也“执行正确”。
- 备选：仅通过文档约定人工检查。
- 未选原因：不可重复、不可自动化。

## 风险 / 权衡

- [风险] SLO 阈值初始设定不合理导致频繁降级 → 采用灰度命名空间先试运行并逐步收敛阈值。
- [风险] 新增治理语义增加认知负担 → 通过固定三态映射与标准化 runbook 降低理解成本。
- [风险] 告警升级过多导致值班疲劳 → 引入最小告警集与抑制窗口，按级别去重。
- [风险] 预算窗口滞后可能延迟阻断 → 保留硬阻断通道（严重错误直接 block）。

## 迁移计划

1. 先定义 SLO 契约与字段，不改变现有发布默认策略。
2. 在预生产环境启用 SLO 评估与三态映射，只输出建议不阻断。
3. 完成闭环演练后切换为“建议 + 条件阻断”。
4. 达到稳定窗口后在生产启用“越线自动降级/阻断”。

回滚策略：若出现误阻断，回退到“仅质量门禁阻断 + SLO 观测告警”模式，保留审计输出但取消自动阻断动作。

## 开放问题

- 错误预算单位是否统一为“动作失败率”，还是按“对象级失败数 + 影响权重”组合计算？
- `degrade` 升级到 `block` 的最小连续窗口应为 1 个还是 2 个？
- 值班通知通道先使用现有告警规则扩展，还是单独维护 `incident-readiness` 规则集？
