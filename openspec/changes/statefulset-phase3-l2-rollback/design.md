## 上下文

当前系统已具备 Deployment 的 L2 健康版本回滚与 StatefulSet 的 Phase 2 受控 L1 动作，但 StatefulSet 缺少 L2 自动回滚，导致 L1 失败后直接降级人工，恢复时延与操作一致性不足。Phase 3 需要在既有“安全优先”框架下补齐 StatefulSet 的 L2，同时复用现有快照、门禁、审计与指标机制，避免引入平行实现。

## 目标 / 非目标

**目标：**
- 为 StatefulSet 增加 L2 健康版本回滚能力，形成 L1→L2→L3 渐进式闭环。
- L2 执行前必须完成快照、候选健康版本筛选与依赖完整性校验。
- L2 失败必须执行恢复与冻结，回退只读并升级人工介入。
- 复用统一编排入口与观测维度，保证 Deployment/StatefulSet 行为一致可检索。

**非目标：**
- 本阶段不引入跨命名空间批量回滚。
- 本阶段不实现自动审批服务或外部策略引擎。
- 本阶段不调整 Deployment 既有 L2 行为语义，仅做多工作负载统一扩展。

## 决策

1. **统一编排状态机扩展，而非 StatefulSet 专用编排器**
   - 方案 A：新增 StatefulSet 专用 orchestrator。
   - 方案 B：在现有 orchestrator 增加 workload-aware L2 分支。
   - 选择 B：减少实现分叉，复用已有幂等、熔断与审计逻辑。

2. **L2 候选采用“可验证健康版本”而非“最近版本”**
   - 方案 A：直接回滚到最近历史版本。
   - 方案 B：只允许回滚到满足稳定窗口与依赖校验的健康候选。
   - 选择 B：降低把系统回滚到不可用版本的风险。

3. **L2 失败后进入冻结窗口并强制人工解锁**
   - 方案 A：自动重试直到熔断。
   - 方案 B：失败一次进入冻结，只读阻断后续自动动作。
   - 选择 B：符合保守策略，避免 StatefulSet 有序副本反复扰动。

4. **配置驱动阈值与候选窗口**
   - 方案 A：内置固定阈值。
   - 方案 B：通过 CRD/values 声明候选窗口、冻结时长、灰度阈值。
   - 选择 B：满足禁止硬编码与可灰度要求。

## 风险 / 权衡

- [风险] 候选筛选过严导致频繁降级 L3 → [缓解] 提供可配置观察窗口并输出筛选证据用于调参。
- [风险] 回滚恢复链复杂度增加 → [缓解] 统一快照恢复路径并强化失败路径单测。
- [风险] StatefulSet 依赖校验覆盖不足 → [缓解] 在设计中约束最小依赖集（ConfigMap/Secret/PVC 绑定状态）并留扩展点。
- [风险] 生产误启导致超预期动作 → [缓解] 默认关闭 L2，白名单 + 审批 + 指标门禁三重约束。

## Migration Plan

1. 增加 Phase 3 配置字段与状态字段（候选证据、L2 结果、降级建议）。
2. 在 staging 以单命名空间开启 StatefulSet L2 灰度，执行成功/失败/恢复演练。
3. 验证关键指标阈值（成功率、回退率、冻结触发率）连续满足后分批扩大范围。
4. 若指标越线或出现误动作，关闭 L2 开关并保持 L1+L3 路径。
5. 形成发布检查清单与回滚手册，进入生产逐步放量。

## Open Questions

- StatefulSet 候选版本“健康”是否额外要求 PVC 绑定稳定窗口。
- L2 回滚后是否需要强制最短观察期才允许再次进入 L1/L2。
- 人工解锁冻结的最小权限模型与审计字段是否需要独立 capability。
