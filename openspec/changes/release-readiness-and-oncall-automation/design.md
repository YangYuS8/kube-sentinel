## 上下文

当前 Kube-Sentinel 在策略执行、质量门禁和契约治理层已经形成较完整链路，但“发布窗口内值班是否能快速判断并执行动作”仍存在认知成本。门禁输出、闭环演练证据、值班 runbook 和发布审批信号尚未完全统一，导致在高压窗口下需要人工跨多个输出面板进行拼接判断。

本设计聚焦把“发布就绪判定”与“值班动作”标准化为单一可审计输出，并与现有质量门禁、闭环演练、生产门禁和可观测能力衔接，避免引入新的策略复杂度。

## 目标 / 非目标

**目标：**
- 定义统一发布就绪摘要模型：必须输出 allow/degrade/block、关键原因与下一步动作。
- 定义值班动作模板映射：告警等级、runbook、审批触发点、回滚路径必须可程序化关联。
- 将闭环演练结果汇总为发布可用报告：支持发布前一键核对。
- 将发布就绪度指标和告警映射纳入可观测规范，形成持续追踪能力。

**非目标：**
- 不新增 L4/L5 等新的自愈策略层级。
- 不重构 controller 主执行链路。
- 不引入新的外部编排系统或审批平台。

## 决策

### 决策 1：新增独立能力 `release-readiness-and-oncall-automation`
- 选择：将发布就绪判定与值班自动化要求放在独立能力中。
- 原因：该能力跨越质量门禁、运行时闭环和运维流程，独立能力更利于演进。
- 备选：散落到现有能力中会加剧语义分散，不利于值班执行。

### 决策 2：现有能力采用增量约束而非重写
- 选择：对 `runtime-closed-loop-validation`、`delivery-quality-gates`、`runtime-production-gating`、`observability` 做 ADDED 增量。
- 原因：最小变更面，降低同步和归档风险。
- 备选：整篇重写会引入高噪声 diff，不利于审阅。

### 决策 3：发布判定采用“摘要 + 证据明细”双层结构
- 选择：摘要层用于值班快速决策，明细层用于追溯和审计。
- 原因：兼顾决策效率与治理可审计性。
- 备选：仅输出明细会增加值班读取成本；仅输出摘要会丢失诊断深度。

### 决策 4：值班动作模板化并与门禁等级强绑定
- 选择：将 `allow/degrade/block` 映射到固定动作模板（通知级别、runbook、审批、回滚）并要求演练校验。
- 原因：降低人为解释偏差，提升跨班组一致性。
- 备选：自由文本建议虽灵活但难以自动化验收。

## 风险 / 权衡

- [风险] 输出字段增多可能影响可读性  
  → 缓解：明确“摘要必读字段”与“扩展证据字段”分层展示。

- [风险] 模板化动作在极端场景下不够灵活  
  → 缓解：保留人工 override，但要求记录原因和复盘字段。

- [风险] 门禁和演练之间口径不一致  
  → 缓解：在任务中加入一致性测试与对照断言。

## Migration Plan

1. 规范阶段：明确摘要字段、值班模板和验收矩阵。  
2. 实现阶段分三批：
   - 批次 A：质量门禁与生产门禁输出发布摘要；
   - 批次 B：闭环演练输出值班模板与自动汇总；
   - 批次 C：可观测指标与文档收口。  
3. 先在预生产启用自动汇总与模板校验，再推广到生产值班流程。  
4. 如出现误判，临时降级到“摘要告警 + 人工裁决”模式并保留证据采集。

## Open Questions

- 发布就绪摘要是否需要单独持久化对象（例如 CR 状态字段外的报告文件）？
- 值班模板中的审批触发点是否要按命名空间/业务等级细分？
- 是否需要为“连续 degrade 但未 block”的长尾场景定义专门策略模板？
