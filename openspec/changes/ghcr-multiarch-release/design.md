## 上下文

当前 Kube-Sentinel 的交付形态以“源码仓库 + 本地构建 + 本地集群导入”为主。现有 [README]、安装脚本和 chart 已经支持通过外部镜像引用部署控制器，但仓库尚未定义官方镜像从何处发布、何时发布、如何区分稳定版与预发布，也没有将多架构构建纳入正式分发链路。

这导致两个实际问题。第一，外部使用者在没有本地构建环境的情况下缺少可直接消费的官方镜像。第二，随着 arm64 开发机和 amd64 CI/集群并存，多架构验证会继续依赖操作者各自构建，难以形成可回滚、可审计的发布基线。

## 目标 / 非目标

**目标：**

- 为 Kube-Sentinel 定义单独的官方控制器镜像发布链路，不与现有 CI 校验或夜间演练职责混淆。
- 将正式镜像发布入口固定为 `v` 前缀语义化版本 tag push，并支持稳定版与预发布版的差异化标签行为。
- 发布单个可消费的 GHCR 镜像引用，并要求其同时包含 `linux/amd64` 与 `linux/arm64` manifest。
- 保留现有本地构建与 minikube 导入路径，使官方预构建镜像成为新增能力而非替代路径。
- 使发布结果能够直接被 README、安装说明和 chart 默认镜像策略消费。

**非目标：**

- 不在本变更内实现 Helm chart 发布、GitHub Release 自动生成、SBOM 签名或 provenance 证明。
- 不将普通 `push main` 扩展为自动镜像发布入口。
- 不引入新的外部制品仓库，官方镜像分发仅覆盖 GHCR。
- 不修改控制器自愈逻辑、CRD 语义或运行时门禁行为。

## 决策

### 决策 1：正式镜像发布与现有 CI 分离，采用独立 workflow

- 选择：新增一条专用镜像发布工作流，不复用当前 `ci` 与 `nightly-delivery-drill` 工作流。
- 原因：现有工作流职责分别是“合并验证”和“周期演练”，而镜像发布属于制品分发；混用会把发布权限、失败语义和触发条件耦合到日常校验里。
- 备选：在现有 `ci` 中增加条件分支会让工作流同时承担校验与制品发布，后续更难维护和审计。

### 决策 2：正式发布入口固定为 `v` 前缀语义化版本 tag push

- 选择：仅接受 `vX.Y.Z` 与 `vX.Y.Z-<prerelease>` 形式的 tag 进入正式发布工作流；普通分支 push 不触发镜像发布。
- 原因：tag 自带版本语义，便于回滚、审计和对外引用，也能满足“避免每次提交都触发构建”的要求。
- 备选：使用 `release.published` 会把版本事实依赖到 GitHub Release UI；使用 `push main` 则会让稳定版标签与开发中提交混在一起。

### 决策 3：官方镜像名采用仓库根路径 ghcr.io/YangYuS8/kube-sentinel

- 选择：将官方运行镜像直接映射到仓库根路径，而不是额外追加 `/controller` 子路径。
- 原因：当前仓库只有一个正式运行镜像，根路径最短、最直观，也更符合“让他人直接部署”的目标。
- 备选：使用 `/controller` 虽然为未来多镜像工件预留层级，但会在当前阶段引入额外命名复杂度，而没有明确收益。

### 决策 4：多架构镜像采用 buildx 构建单一 manifest 列表

- 选择：工作流应使用平台感知构建，将 Dockerfile 从写死 `GOARCH=amd64` 调整为根据目标平台输出对应二进制，并最终推送同时包含 `linux/amd64` 与 `linux/arm64` 的单一镜像引用。
- 原因：对外消费方不应区分两个不同镜像名；单一 manifest 列表能让 amd64 与 arm64 节点自动拉取匹配层，并降低文档和安装复杂度。
- 备选：分别维护两个独立镜像标签会增加 chart、文档和回滚矩阵复杂度，也更容易产生版本漂移。

### 决策 5：`latest` 仅代表稳定版，预发布与手动触发不得覆盖

- 选择：当 tag 为稳定版 `vX.Y.Z` 时，发布 `vX.Y.Z` 与 `latest`；当 tag 为预发布如 `vX.Y.Z-rc.1` 时，仅发布该版本标签，不更新 `latest`。
- 原因：`latest` 应保持“对外稳定入口”的语义，不能被 RC、beta 或人工补发结果污染。
- 备选：让所有版本都刷新 `latest` 会破坏可回滚性和外部部署的可预期性。

### 决策 6：`workflow_dispatch` 仅作为运维入口，不拥有正式发布语义

- 选择：保留手动触发用于预发布验证或补发，但其默认行为不得覆盖 `latest`，也不应替代 tag push 成为正式发布事实来源。
- 原因：这能保留恢复和调试空间，同时防止人工操作绕过版本规则。
- 备选：让手动触发与 tag push 等价，会造成正式发布边界不清，增加审计成本。

## 风险 / 权衡

- [风险] 多架构构建会增加发布时长和工作流复杂度
  → 缓解：将多架构构建限定在版本发布工作流，不影响日常 PR 或 main 校验时长。

- [风险] Dockerfile 当前写死 amd64，迁移后若平台参数处理错误会生成错误架构二进制
  → 缓解：在设计与任务中显式要求校验 manifest 平台列表，并用拉取结果验证 amd64/arm64 标签一致性。

- [风险] `latest` 语义如果处理不严，会让预发布结果污染稳定消费路径
  → 缓解：将稳定版和预发布版的标签判定写入规范，并在工作流条件中显式区分。

- [风险] 官方镜像发布后，团队可能误以为本地 build 路径可以删除
  → 缓解：在本地部署能力规范中明确两条路径并存，保留 minikube 本地导入入口。

## Migration Plan

1. 将 Dockerfile 调整为平台感知构建，消除当前写死 amd64 的限制。
2. 新增专用 GHCR 发布工作流，并绑定 `v` 前缀语义化版本 tag push 与受控的 `workflow_dispatch`。
3. 将 README、安装说明和默认镜像配置更新为“本地构建路径 + 官方预构建镜像路径”双入口说明。
4. 使用预发布 tag 验证多架构 manifest、标签行为与 `latest` 保护规则。
5. 在首个稳定版 tag 发布后，才将 `latest` 视为对外稳定入口。
6. 回滚策略：若发布工作流或多架构产物异常，可停止打新 tag，并继续使用现有本地构建/导入路径；必要时撤回默认文档中的官方镜像入口。

## Open Questions

- 首个稳定版发布前，chart 默认值是否仍保持本地镜像名称，还是在文档中优先推荐显式覆盖远端镜像标签？
