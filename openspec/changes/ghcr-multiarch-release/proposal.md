## 为什么

当前 Kube-Sentinel 已具备本地构建、最小安装和 CI 质量门禁能力，但对外分发仍停留在“开发者本地构建镜像”的阶段。随着项目进入更稳定的联调与对外试用阶段，缺少官方可拉取的多架构镜像会同时抬高他人部署成本和本地 minikube 验证成本，尤其是在 arm64 与 amd64 混合环境下更明显。

现在推进这项变更，能够在不改变控制器核心自愈语义的前提下，为发布、验证、回滚和跨架构测试提供统一的镜像分发基线，并把“何时发布、发布什么标签、哪些镜像可作为稳定入口”从隐含约定收敛为显式规则。

## 变更内容

- 增加一条独立于现有 CI 校验与夜间演练之外的官方镜像发布路径，用于构建并发布 Kube-Sentinel 控制器镜像。
- 将官方发布入口固定为 `v` 前缀语义化版本 tag push，避免普通提交或非正式标签触发镜像发布。
- 将官方镜像分发目标固定为 GHCR，并要求单个镜像引用同时支持 `linux/amd64` 与 `linux/arm64` 两种架构。
- 明确稳定版标签、预发布标签与 `latest` 的关系，确保 `latest` 仅代表稳定版发布结果，不被预发布或临时补发覆盖。
- 保留现有本地 build + minikube image load 路径，确保远端镜像分发是新增能力而不是替代本地开发回路。

## 功能 (Capabilities)

### 新增功能

- `container-image-distribution`: 定义 Kube-Sentinel 官方控制器镜像的发布入口、标签语义、多架构分发和稳定版消费约束。

### 修改功能

- `local-deployment-and-dev-loop`: 补充“本地构建导入”和“消费官方预构建镜像”两条路径并存的约束，明确远端分发不替代本地联调入口。

## 影响

- 受影响代码：GitHub Actions 工作流、Dockerfile、多架构构建参数、README、安装说明，以及默认镜像引用相关配置。
- 受影响系统：GHCR 发布链路、版本发布流程、minikube 与其他测试集群中的镜像拉取路径。
- 风险控制：需要显式约束稳定版与预发布标签行为，避免 `latest` 漂移、半成品多架构 manifest 或错误架构二进制进入正式分发。
