## 上下文

当前变更聚焦 v1alpha1 的可实施边界：仅支持 Deployment，但不能把架构锁死在单一工作负载类型；同时需要将回滚目标和熔断治理从“经验判断”提升为可验证规则。现状问题是：策略升级、回滚目标选择和失败扩散控制缺少统一设计，难以保障幂等性、可解释性与安全性。

## 目标 / 非目标

**目标：**
- 建立仅面向 Deployment 的 v1alpha1 自愈主链路，并定义可扩展的 Workload 适配接口。
- 明确 L2 回滚基准为“最近一次健康运行的版本”，并给出基于 Revision 历史的选择流程。
- 设计双层熔断机制（对象级 + 命名空间/全局级），控制局部与系统性风险。
- 让上述决策在状态字段、审计事件和监控指标中可观测、可追溯。

**非目标：**
- 不在 v1alpha1 实现 StatefulSet 的完整执行逻辑。
- 不引入独立可观测平台或复杂多租户策略中心。
- 不在本变更内覆盖跨集群联动治理。

## 决策

### 决策 1：v1alpha1 仅执行 Deployment，自愈执行与对象差异通过适配接口隔离
- 方案：Reconciler 统一编排流程，WorkloadAdapter 负责 workload 特有操作（读取副本、滚动状态、Revision 解析、回滚执行）。
- 理由：短期收敛可交付范围，长期保留 StatefulSet 等扩展路径，避免控制器内出现对象分支爆炸。
- 备选：直接在 Reconciler 内写 if/else 区分对象类型；放弃，因耦合过高且后续演进成本大。

### 决策 2：回滚目标固定为“最近一次健康运行 Revision”
- 方案：按 Revision 时间逆序遍历历史，选择最近且满足健康判定的 Revision 作为 L2 目标。
- 健康判定基线：在观测窗口内无关键告警、可用副本达到期望、无持续重启异常。
- 理由：相比“上一个 Revision”更稳健，能规避刚发布失败版本反复回滚震荡。
- 备选：仅回滚到上一个版本；放弃，因无法保证上一个版本确实健康。

### 决策 3：双层熔断采用“先对象后域级”的分级治理
- 方案：
  - 第一层（对象级）：针对单个 Deployment 记录失败计数与冷却期，达到阈值即暂停该对象自动动作。
  - 第二层（命名空间/全局级）：聚合失败密度与影响面，当超过阈值时暂停该范围内新自愈动作，仅保留告警与人工介入。
- 理由：对象级可快速止损，域级可防止系统性抖动与级联故障。
- 备选：仅单层熔断；放弃，因难以同时覆盖局部和全局风险。

## 风险 / 权衡

- [风险] 健康判定窗口过短导致误判“健康 Revision” → [缓解] 采用最小观察时长与多指标交叉判定。
- [风险] 域级熔断阈值过严影响自愈可用性 → [缓解] 阈值配置化并提供灰度启用。
- [风险] 适配接口过早抽象导致复杂度提升 → [缓解] v1alpha1 只保留最小接口集，按需求增量扩展。

## Migration Plan

1. 先上线仅 Deployment 的主链路，并默认启用对象级熔断。
2. 以配置开关灰度启用域级熔断，先在低风险命名空间验证。
3. 完成 Revision 健康判定指标与审计字段对齐后，启用“最近健康 Revision 回滚”策略。
4. 发现误判或回滚异常时，可回退到“仅 L1 + 人工介入”模式并保留审计数据。

## Open Questions

- “健康运行”是否要求与业务 SLA（如错误率）绑定，还是先采用平台通用健康指标？
- 域级熔断默认作用域应为命名空间还是全局，是否需要按标签分组？
- 是否需要为回滚目标加入人工审批白名单机制（仅生产环境启用）？
