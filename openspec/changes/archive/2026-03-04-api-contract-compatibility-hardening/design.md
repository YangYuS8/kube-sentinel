## 上下文

当前系统已形成运行时门禁、SLO 治理与闭环演练能力，但 API 契约治理仍分散在 CRD 注释、脚本校验和文档约定中，缺少统一的“规范 -> 校验 -> 证据”链路。现状风险是：字段约束变更可能在 CRD、Helm、状态输出之间出现偏差，导致发布阶段才暴露兼容性问题。

本设计目标是在不扩大业务能力范围的前提下，补齐 API 契约层的可持续交付基础：统一字段约束治理、明确兼容性策略、标准化 status 语义，并将其纳入质量门禁证据。

## 目标 / 非目标

**目标：**
- 建立 API 契约治理基线：默认值、可选性、校验约束必须可追溯且可验证。
- 建立 API 兼容性治理流程：变更必须标注兼容级别与迁移路径。
- 统一状态字段最小语义集合：策略阶段、最近动作、失败原因、下一步建议。
- 将 API/CRD/Helm 一致性结果纳入质量门禁与发布证据输出。
- 对齐生产门禁判定证据，使“是否可放量”可由统一证据判定。

**非目标：**
- 不引入新的自愈策略等级或新的业务编排动作。
- 不重构既有控制器主流程。
- 不新增外部依赖系统（数据库、消息中间件等）。

## 决策

### 决策 1：新增 `api-contract-governance` 能力承载契约规则
- 选择：新增独立能力规范，集中描述 API 契约和兼容性要求。
- 原因：避免规则分散到多个现有能力中导致边界不清。
- 替代方案：
  - 仅修改 `runtime-production-gating`：会把 API 契约与发布门禁耦合，长期可维护性差。
  - 仅修改 `delivery-quality-gates`：会把“检查”与“契约定义”混在一起，规范语义不完整。

### 决策 2：对现有能力采用“增量补充”而非重写
- 选择：在 `delivery-quality-gates` 与 `runtime-production-gating` 中以增量需求补齐契约证据要求。
- 原因：保持变更最小化，降低归档同步风险。
- 替代方案：整体改写主规范，成本高且容易引入无关变动。

### 决策 3：兼容性策略采用显式分类
- 选择：API 变更必须分类为 backward-compatible / migration-required / version-bump-required，并给出迁移路径。
- 原因：提升评审与发布决策一致性，避免“默认兼容”假设。
- 替代方案：自由文本描述兼容性，信息不结构化，难以门禁化。

### 决策 4：status 语义采用最小强约束集
- 选择：要求稳定输出 phase、lastAction、failureReason、nextRecommendation 四类语义。
- 原因：直接满足配置中的 API contract 约束，且与现有闭环演练证据天然对齐。
- 替代方案：按对象类型分别定义，短期灵活但长期对齐成本高。

## 风险 / 权衡

- [风险] 规范新增约束可能提升短期实现与评审成本  
  → 缓解：采用增量规范，先约束“最小必须集合”，避免一次性过度扩展。

- [风险] 兼容性分类落地口径不一致  
  → 缓解：在 tasks 中加入示例化验收标准与边界用例。

- [风险] status 最小语义与既有字段命名存在局部差异  
  → 缓解：允许兼容映射，但必须在规范中定义语义对齐规则。

- [风险] 发布门禁证据字段增多导致可读性下降  
  → 缓解：在门禁输出中分层展示（判定结果层 / 契约证据层）。

## Migration Plan

1. 先完成提案与增量规范：确定能力边界、兼容性分类与验收口径。  
2. 进入实现阶段后按任务分三批推进：
   - 批次 A：API 契约与 status 语义校验落地；
   - 批次 B：质量门禁输出与 CI 证据绑定；
   - 批次 C：文档、回滚与灰度发布说明收口。  
3. 发布采用灰度启用：先在预生产验证兼容性判定，再推进生产。  
4. 若发现误阻断，优先回退到“仅告警不阻断”模式，再修正契约规则。

- 兼容性分类是否需要额外定义“时间窗内允许双写”的迁移策略模板？
