## 上下文

当前仓库已经完成 Deployment L1 的首发闭环，并通过了质量门禁与 minikube smoke，但安装与本地验证仍分散在多个手工步骤中：CRD 需要单独安装，本地 manager 需要独立启动，minikube 代理与 blast radius 需要操作者自行判断，Helm chart 目前也更偏配置契约而不是完整安装入口。这使得后续每一条能力变更都需要重复搭环境、排查误报，并抬高评审与灰度验证成本。

## 目标 / 非目标

**目标：**

- 提供一个最小可安装的部署入口，使测试环境不再依赖纯手工 `go run ./cmd/manager` 流程。
- 提供一个面向开发者的本地开发回路，统一环境自检、CRD 安装、manager 启动、demo workload 与 smoke 验证。
- 将 minikube 代理、端口占用、低 Pod 基数导致的 blast radius 阻断等常见环境问题前移为显式检查与提示。
- 让后续功能切片都能复用同一条本地与灰度验证路径，降低回归成本。

**非目标：**

- 不在此变更中实现 Deployment L2/L3 自动升级。
- 不在此变更中扩展 StatefulSet 自动写路径。
- 不在此变更中重构现有 HealingRequest API 语义或门禁算法。
- 不追求一次完成完整生产级安装体系；优先交付最小可重复部署与开发验证入口。

## 决策

### 决策 1：优先补“最小可安装入口”，而不是继续增加处置策略

- 选择：先解决部署与本地验证摩擦，再继续扩展 L2/L3 或 StatefulSet 能力。
- 原因：当前系统的主要风险不是能力空白，而是每次验证都要重复人工搭环境；如果不先收敛开发回路，后续每条能力变更的真实验证成本都会继续上升。
- 备选方案：
  - 直接推进 Deployment L2：能增加产品能力，但会叠加更高的验证复杂度。
  - 直接推进 StatefulSet Phase 2：能扩大工作负载覆盖，但会把部署与调试摩擦同步放大。

### 决策 2：部署入口分为“本地开发入口”和“集群安装入口”两层

- 选择：一层提供开发者友好的本地脚本/命令入口，另一层提供可在测试环境复用的安装打包入口。
- 原因：本地调试与集群安装面对的约束不同，强行用单一路径会导致要么开发效率低，要么安装入口不清晰。两层入口可以共享同一套 CRD、默认配置和 smoke 验证，但职责分离更清楚。
- 备选方案：
  - 只做 Helm 安装入口：对测试环境友好，但本地调试仍需要拼接很多手工步骤。
  - 只做 dev script：能提速本地调试，但无法显著改善测试环境的安装一致性。

### 决策 3：把环境误报前移为显式自检，而不是留给故障日志解释

- 选择：在本地开发回路中显式检查 kube context、CRD 是否安装、端口占用、minikube 基础组件状态，以及小集群 blast radius 风险。
- 原因：当前最常见的问题不是业务逻辑错误，而是环境没准备好。把这些检查前移，可以减少“功能看起来坏了，实际只是环境没就绪”的排查成本。
- 备选方案：
  - 仅靠 README 说明：信息存在，但操作者仍要自己拼接判断。
  - 仅靠 manager 启动日志：能解释失败，但不能阻止用户在错误环境里继续联调。

### 决策 4：smoke 验证固定为阻断路径与成功路径两类

- 选择：本地开发回路内置两类最小验证：
  - 默认配置下验证 gate/block 语义
  - 放宽单次请求 blast radius 后验证 PendingVerify -> Completed 成功闭环
- 原因：这两条路径能够覆盖当前 MVP 最核心的真假阳性判断，比一次性堆积更多场景更适合日常迭代。
- 备选方案：
  - 只验证成功路径：容易忽略默认安全策略是否仍然生效。
  - 一次覆盖所有场景：脚本更复杂，日常反馈反而变慢。

## 风险 / 权衡

- [风险] 安装入口新增后，Helm/default 配置与本地脚本可能漂移。
  → 缓解：让开发脚本尽量复用 chart 默认值与现有 CRD/quality-gate 检查。
- [风险] 为了方便本地 smoke 放宽 blast radius，可能让用户误以为生产也应这样配置。
  → 缓解：明确要求仅对单次测试请求临时调整，并在文档中强调恢复保守阈值。
- [风险] 最小安装入口如果过度追求完整，可能把提案范围拉大。
  → 缓解：只承诺“测试环境可安装 + 本地开发可复现”，不在本变更中解决全部生产化问题。
- [风险] minikube 与代理问题依赖宿主机环境，无法完全通过代码消除。
  → 缓解：将代理前提、自检结果与失败分类写入脚本与文档，优先降低误判而不是承诺完全自动修复。

## Migration Plan

1. 保持现有 `go run ./cmd/manager` 路径可继续使用，避免影响当前开发者工作流。
2. 新增最小安装入口与本地开发入口，并在 README 中将其设为推荐路径。
3. 将 smoke 验证脚本与闭环检查清单对齐，确保默认阻断与放宽后成功这两类路径都可重复执行。
4. 通过 `make quality-gate`、CRD 一致性检查与本地 minikube smoke 验证新入口。
5. 若新入口引入问题，可回退到现有手工流程，同时保留 CRD、manager 和 webhook 的当前使用方式。
