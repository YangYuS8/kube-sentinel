## 上下文

上一轮变更已完成 v1alpha1 核心策略与门禁规范，但系统仍偏“离线设计可行”，缺少真实运行时闭环验证。主线A目标是以最小范围打通真实事件输入、控制器执行、状态回写与观测输出，证明系统在集群中可运行且行为可解释。

## 目标 / 非目标

**目标：**
- 建立 Alertmanager Webhook 到 HealingRequest 的可追踪接入路径。
- 形成 MVP 级运行时闭环：事件触发、门禁判定、L1/L2/L3推进、状态与审计输出。
- 在真实运行链路中验证三项硬约束：仅 Deployment 写操作、无健康 Revision 降级 L3、双层熔断有效阻断。
- 形成可重复的闭环验收脚本与观测检查清单。

**非目标：**
- 不在本变更中扩展 StatefulSet 执行逻辑。
- 不引入复杂策略引擎重构或跨集群联动。
- 不解决大规模性能调优（吞吐、分片调度等）。

## 决策

### 决策 1：事件入口采用“Webhook Receiver + 映射器”
- 方案：接收 Alertmanager payload 后映射为统一事件模型，再投递/更新 HealingRequest。
- 理由：隔离告警格式与领域对象，避免 Reconciler 直接耦合外部结构。
- 备选：Reconciler 直接解析 webhook 内容；放弃，因职责混杂且难测。

### 决策 2：闭环验收采用“强断言三件套”
- 方案：每次演练必须断言：
  1) 非 Deployment 输入只读拒绝；
  2) 无健康 Revision 必须 L3；
  3) 熔断触发后自动写操作被阻断。
- 理由：与安全门槛直接对齐，能最小成本验证核心可信性。

### 决策 3：观测采用“同一关联键贯穿”
- 方案：以 request UID/trace key 贯穿 status、event、metric、audit。
- 理由：确保问题定位可追踪，减少“看到告警但找不到执行链路”的运维盲区。

## 风险 / 权衡

- [风险] Webhook 噪声或重复告警导致动作抖动 → [缓解] 使用幂等键与速率限制窗口。
- [风险] 健康证据不足导致频繁降级 L3 → [缓解] 明确证据缺失原因并输出可运维提示。
- [风险] 熔断阈值配置不当影响可用性 → [缓解] 默认保守阈值 + 可配置 + 灰度启用。

## Migration Plan

1. 先上线 Webhook 接入与只读映射，验证事件可达与审计可见。
2. 启用闭环执行链路（门禁→策略→状态回写），但保持保守阈值。
3. 通过演练脚本完成三件套断言，再放开生产环境灰度。
4. 若异常，立即降级为“只读评估+告警+人工介入”。
