## 上下文

当前系统已经具备 Deployment/StatefulSet 的分层处置、熔断、快照恢复与闭环演练能力，但生产发布决策仍然分散在多个指标与脚本输出中，缺少统一的“可执行门禁判定器”。这导致上线阶段存在三类风险：
- 指标可见但无法自动阻断，依赖人工主观判断。
- 熔断、安全门禁、发布门禁并发命中时缺少稳定优先级。
- 发生降级后缺少统一证据结构，不利于快速回溯与值班协同。

本设计聚焦“运营化上线门禁”，把现有能力编排为统一流程：信号收集 → 门禁判定 → 自动降级/阻断 → 可追溯证据输出。

## 目标 / 非目标

**目标：**
- 定义统一门禁判定模型（allow / block / degrade）及其输入指标。
- 明确门禁优先级（维护窗口/安全门禁 > 熔断 > 发布门禁阈值）。
- 定义越线后的标准动作（切换保守模式、只读评估、人工介入建议）。
- 统一输出门禁证据（原因码、阈值快照、关键指标、建议动作）。
- 将门禁流程纳入预生产与小流量灰度验收矩阵。

**非目标：**
- 不引入新的工作负载类型或跨集群编排。
- 不引入外部策略引擎（如 OPA/CEL 服务化托管）。
- 不在本次变更中重构 Alertmanager 接入协议。

## 决策

1. **采用“单判定器 + 分层优先级”而非分散判断**
   - 方案 A：每个模块独立判断门禁并返回局部结论。
   - 方案 B：集中到统一门禁判定器，按优先级输出唯一结论。
   - 选择 B：降低冲突与重复逻辑，便于演练与审计。

2. **采用“阻断原因码”作为跨模块契约**
   - 方案 A：仅依赖自然语言日志。
   - 方案 B：标准化 reason code + 阈值快照 + 指标上下文。
   - 选择 B：便于告警聚合、报表统计与自动化回退。

3. **采用“保守模式优先回退”而非继续尝试自动修复**
   - 方案 A：门禁越线后继续重试更高等级动作。
   - 方案 B：越线即降级到保守模式并强制只读评估。
   - 选择 B：符合安全优先和可回滚优先原则。

4. **采用“两阶段验证”作为首版上线策略**
   - 方案 A：直接生产全量启用。
   - 方案 B：预生产全路径演练 + 小流量灰度观测再扩展。
   - 选择 B：降低误阻断和误放行风险。

## 风险 / 权衡

- [阈值不合理导致误判] → 采用白名单灰度 + 逐步收敛阈值 + 保守默认。
- [并发门禁冲突] → 强制统一优先级与唯一判定输出。
- [指标延迟导致滞后决策] → 在判定报告中输出样本窗口与数据完整性标识。
- [降级后恢复不及时] → 明确恢复条件与人工确认步骤，并输出下一步建议。

## 迁移计划

1. 新增 `runtime-production-gating` 规范并定义判定契约。
2. 将既有 capability 对齐统一判定契约（hardening、validation、breaker、tiered-healing、rollback）。
3. 在预生产环境执行完整演练：允许、阻断、降级三类路径。
4. 以命名空间白名单进行小流量灰度并观察关键指标窗口。
5. 达到阈值稳定后扩展范围；越线则一键回退到保守模式。

## Open Questions

- 是否需要在后续版本引入“人工批准后一次性放行”机制？
