## 上下文

当前系统已具备基础闭环能力，但缺少“绝不误动作”导向下的统一决策框架：

- L2 回滚前置证据门槛不够显式，部分路径对证据完整性的约束不足。
- 门禁阻断虽已存在，但缺少命名空间级预算与可解释的保守降级语义。
- 告警输入存在抖动与瞬时毛刺，缺少标准化观察期分级矩阵。

该设计面向保守策略落地：在不改变“渐进式处置”主流程前提下，引入更严格的证据与预算机制，优先保障安全与可解释性。

## 目标 / 非目标

**目标：**

- 为 L2 回滚建立三位一体证据门槛：回滚可达性、告警稳定窗口、依赖完整性。
- 引入按应用数计算的命名空间阻断预算，支持最小基数保护。
- 建立告警观察期分级矩阵与 `PendingVerify` / `Suppressed` 状态语义。
- 在阻断场景提供“影子执行说明”，确保审计与运维可解释。
- 定义保守模式验收指标：误放行率、误阻断率、重复执行率、恢复耗时、证据链完整度。

**非目标：**

- 不改变 L1/L2/L3 分级策略的大方向。
- 不在本次设计中引入新的外部数据存储系统。
- 不覆盖跨集群联动与多租户全局仲裁。

## 决策

### 决策 1：L2 采用三位一体证据门槛（MUST）

- 选择：L2 执行前必须同时满足：
  1. 回滚可达性：目标 Revision 健康可用且达到稳定时间窗口。
  2. 告警稳定窗口：按告警分级执行 Soak Time，观察期内自动恢复则抑制。
  3. 依赖完整性：目标 Revision 关联配置（ConfigMap/Secret）存在且可用。
- 备选：仅校验历史 Revision 健康。
- 取舍原因：单一证据在抖动与配置漂移场景下误动作风险高，三重校验可显著降低误回滚概率。

### 决策 2：告警观察期采用分级矩阵（MUST）

- 选择：按告警类别/严重度定义非线性观察期（如 CrashLoop 更短，ProbeFailure 更长）。
- 备选：统一固定观察期。
- 取舍原因：统一窗口要么响应慢要么抗抖动差；分级矩阵可在“响应速度/抗抖动”之间平衡。

### 决策 3：命名空间预算按应用数计算（MUST）

- 选择：以 `UnhealthyWorkloads / TotalWorkloadsInNamespace` 计算命名空间风险比例，触发区域只读阻断。
- 备选：按 Pod 数计算。
- 取舍原因：Pod 数易被单大应用放大而误判；按应用数更能识别基础设施级故障扩散。

### 决策 4：默认保守阻断并输出影子执行说明（MUST）

- 选择：证据不足或预算超限时，统一走只读阻断，同时记录“本应执行动作 + 被阻断原因”。
- 备选：静默阻断或仅日志记录。
- 取舍原因：影子执行说明是可运维与可审计的关键，能降低“过度保守不可解释”的治理成本。

### 决策 5：白名单应急尝试权受硬约束（SHOULD）

- 选择：允许 `criticality: high` 工作负载拥有受限紧急尝试权，失败后立即回归阻断。
- 备选：不提供白名单例外。
- 取舍原因：在核心业务中完全禁止自动动作可能不可接受；受限例外能平衡连续性与安全。

## 风险 / 权衡

- [风险] 过度保守导致阻断率上升、自动修复收益下降  
  → 缓解：设置分层预算、最小基数保护、白名单受限尝试权，并灰度观察误阻断率。

- [风险] 观察期过长导致恢复时间变慢  
  → 缓解：采用分级观察期矩阵，对高严重度告警使用更短窗口。

- [风险] 证据链校验增加运行复杂度与判定延迟  
  → 缓解：并行化证据采集、失败快速短路、观测指标化并持续调优阈值。

- [风险] 白名单被滥用削弱保守策略  
  → 缓解：白名单尝试次数、时间窗、审批来源与审计字段全部强约束。

## 迁移计划

1. 在低风险命名空间启用保守模式（仅观察与影子执行说明）。
2. 启用分级观察期与三位一体证据门槛，保持 L2 写操作默认受控。
3. 启用命名空间预算阻断与最小基数保护，验证区域级阻断行为。
4. 灰度开放白名单紧急尝试权（如需要），并监控失败回退路径。
5. 扩展到核心命名空间前，确认验收指标达到门槛；不达标则回退到只读评估。

回滚策略：关闭保守增强开关，恢复基础门禁与现有分级策略，同时保留审计和事件输出。
