## 上下文

当前系统已具备 SLO 治理模型、门禁三态输出和基础事故响应语义，但在“发布运营”层面仍偏静态：阈值更新缺少节奏控制、灰度放量与回退条件未形成标准流程、告警抑制策略与值班动作之间存在执行偏差。该变更聚焦运营化，不新增核心自愈算法，而是强化“策略启用—观测—调优—复盘”循环。

## 目标 / 非目标

**目标：**
- 将 SLO 能力转化为可执行灰度流程（分层启用、放量/回退门槛、证据留存）。
- 建立阈值调优流程（基线值、观察窗口、审批记录、回滚触发）。
- 强化告警抑制与值班链路，使门禁语义与运行响应一致。
- 在闭环演练中验证“越线 -> 处置 -> 调优 -> 回归”的端到端稳定性。

**非目标：**
- 不引入新的 L1/L2/L3 策略动作。
- 不新增外部事件平台深度集成（如新工单系统）。
- 不修改现有 CRD 版本策略，仅在现有能力上补运营语义与流程约束。

## 决策

1. 使用“分层灰度启用”替代一次性全局启用。
- 方案：先低风险命名空间，再核心命名空间，最后全局；每层都有停机/回退条件。
- 原因：降低误配置与误阻断风险。
- 备选：全局一次性切换。
- 未选原因：爆炸半径过大，回滚成本高。

2. 阈值调优采用“固定观察窗口 + 审批记录”机制。
- 方案：每次调优需要最小观察窗口与审批备注，且可回滚到上一版本阈值。
- 原因：防止频繁改阈值导致判定抖动。
- 备选：按值班即时判断临时修改阈值。
- 未选原因：不可追踪、不可复盘。

3. 门禁与告警语义绑定并增加抑制策略。
- 方案：`allow/info`、`degrade/warning`、`block/critical` 统一映射；同对象同级告警在窗口内去重。
- 原因：避免语义不一致与告警风暴。
- 备选：维持各模块独立告警级别。
- 未选原因：值班难以判断优先级。

4. 复盘闭环作为发布前必做演练项。
- 方案：每轮灰度至少执行一次“越线->回退->调优->恢复”演练并沉淀证据。
- 原因：将运营经验转化为制度化资产。
- 备选：仅在真实故障后复盘。
- 未选原因：被动、反馈慢。

## 风险 / 权衡

- [风险] 灰度阶段过长影响交付速度 → 通过环境分层和并行观察缩短总周期。
- [风险] 告警抑制过强导致漏报 → 保留 critical 告警不抑制，仅对 warning 去重。
- [风险] 阈值调优频率过高导致团队疲劳 → 设定最小调优间隔和审批门槛。
- [风险] 复盘流程执行不一致 → 固化模板字段并纳入验收清单。

## 迁移计划

1. 先补流程与文档约束，默认仅观测不改变已有阻断策略。
2. 在预生产环境启用灰度分层与调优流程校验。
3. 通过闭环演练后，在生产启用“受控放量 + 条件回退”。
4. 连续稳定窗口后再放宽人工审批频率。

回滚策略：若出现频繁误阻断或告警风暴，回退到“固定阈值 + 基础门禁 + 手动复盘”模式，暂停自动调优流程。

## 开放问题

- 告警抑制窗口默认值应按对象固定，还是按严重级别差异化？
- 阈值调优审批最小信息集是否需要包含业务影响评分？
- 灰度层级切换由手动触发还是按自动准入条件触发？
