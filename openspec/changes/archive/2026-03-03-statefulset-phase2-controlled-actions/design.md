## 上下文

Phase 1 已将 StatefulSet 纳入保守只读链路，建立了接入、门禁、审计、事件和指标的统一语义，但自动处置仍完全依赖人工。为满足项目目标中的“Deployment/StatefulSet 核心对象优先覆盖”，需要进入 Phase 2：在强约束条件下开放有限自动动作，并保持默认安全姿态。

## 目标 / 非目标

**目标：**

- 在默认关闭前提下，为 StatefulSet 引入受控自动动作能力（先单对象、单次、可回滚）。
- 建立“授权 + 白名单 + 审批 + 运行态证据”四重门禁，任一不满足即回到只读阻断。
- 复用现有保守状态机与审计框架，确保与 Deployment 语义一致且可检索。
- 提供清晰灰度、回滚与发布阻断指标。

**非目标：**

- 不在本次开放多步链路自动升级（L1→L2 全自动串行升级）。
- 不在本次支持跨命名空间批量动作。
- 不引入新的外部控制面依赖（如额外审批系统服务）。

## 决策

1. **动作范围先收敛到“L1 受控动作”**
   - 方案 A：同时开放 L1 与 L2。
   - 方案 B：先开放 L1，L2 仍需人工确认。
   - 选择 B，降低误动作代价并保留可回滚空间。

2. **显式授权模型而非隐式推断**
   - 方案 A：根据风险评分自动判断是否可写。
   - 方案 B：配置开关 + 命名空间白名单 + 对象级审批标记。
   - 选择 B，符合“无法证明安全即人工介入”的原则。

3. **失败即退回只读并冻结重试窗口**
   - 方案 A：失败后自动重试直到熔断。
   - 方案 B：失败一次即退回只读，并要求人工解锁。
   - 选择 B，避免 StatefulSet 高风险重复扰动。

4. **多工作负载统一策略接口，分工作负载实现细节**
   - 方案 A：StatefulSet 独立编排器。
   - 方案 B：统一 orchestrator，workload adapter 扩展差异化检查。
   - 选择 B，减少分叉逻辑与长期维护成本。

## 风险 / 权衡

- [风险] Phase 2 配置误开导致超预期自动动作 → [缓解] 默认关闭、配置校验、审批必填与只读回退。
- [风险] StatefulSet 有序副本特性导致动作副作用放大 → [缓解] 限制单次动作范围并增加副本健康前置检查。
- [风险] 观测指标不足导致灰度判断失真 → [缓解] 增加 workloadKind/actionType/decision 三维指标与发布阈值。
- [风险] 运行态证据波动导致策略抖动 → [缓解] 引入稳定采样窗口与冻结期。

## Migration Plan

1. 引入 Phase 2 配置字段与 CRD 状态字段，默认禁用可写动作。
2. 在 staging 命名空间开启白名单与审批标记，执行演练脚本验证成功/失败/回退路径。
3. 观测连续窗口满足阈值后，小流量灰度到生产低风险命名空间。
4. 如出现误动作或指标越线，立即关闭开关并自动回退只读策略。
5. 保留人工介入链路作为长期兜底，不允许绕过。

## Open Questions

- 审批标记放在注解还是 status.conditions 更便于审计追踪。
- L1 受控动作是否应强制最小冷却时间，避免短时重复扰动。
- Phase 3（可能开放 L2）的前置证据门槛是否需要单独 capability。
