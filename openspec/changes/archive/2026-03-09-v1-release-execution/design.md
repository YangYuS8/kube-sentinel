## 上下文

Kube-Sentinel 当前已经具备 Deployment 首发闭环所需的大部分基础能力：Webhook 接入、HealingRequest 编排、安全门禁、快照与回滚、审计与指标、统一质量门禁、多架构镜像发布、go-live 与 pilot/cutover 判定样例。问题不在于核心能力缺失，而在于这些能力仍分散在 README、发布文档、演练样例、工作流和脚本之中，尚未被收束成首个可交付版本的统一发布执行闭环。

这会带来两个风险：一是团队容易继续扩展非首发边界能力，推迟真正的版本交付；二是即使代码已经满足首发范围，也缺少一个可重复执行、可留证据、可回滚的正式发布路径。这个变更的目标是把“已有能力”转化为“可发布版本”。

## 目标 / 非目标

**目标：**

- 固化首发版本边界，明确 Deployment 自动闭环是 v1 首发交付对象。
- 定义从预生产演练、RC 镜像、pilot/cutover 到稳定版 tag 的统一发布顺序。
- 将现有 install、dev-loop、smoke、delivery pipeline、release-image workflow 串联为一条首发发布执行链。
- 明确首发所需的最小证据包、发布记录和回滚基线。

**非目标：**

- 不扩展新的核心自愈能力，不新增 StatefulSet 自动写动作验收范围。
- 不引入新的镜像仓库、发布平台或独立发布控制平面。
- 不将 Headlamp、Grafana 或更多 UI 能力作为首发阻断前提。

## 决策

### 决策 1: 新增一个面向首发发布的统领 capability

选择新增 `v1-release-execution` capability，专门描述首发版本的发布边界、执行顺序、证据要求和回滚基线。

备选方案：

- 仅继续在 `v1-go-live-readiness` 和 `v1-pilot-readiness-and-cutover` 中补细节。

不采用的原因：

- 这样会让“发布闸门规则”和“首发版本执行流程”继续混在一起，难以清晰表达首发版本的边界和步骤。

### 决策 2: 首发发布流程固定为“预生产验证 -> RC -> pilot/cutover -> 稳定版”

首发版本必须先完成预生产全链路演练，再发布 RC 镜像，随后进入 pilot/cutover 观察，最后才允许发布稳定版标签与 latest。

备选方案：

- 直接从当前 main 分支发布稳定版镜像。
- 先发稳定版，再用 pilot/cutover 补验证。

不采用的原因：

- 会削弱现有 go-live 和 pilot/cutover 规格的意义，也会把首发风险前移到正式版本。

### 决策 3: 复用现有脚本和工作流，不新增并行发布入口

首发发布执行必须复用现有的 `install-minimal`、`dev-local-loop`、`drill-runtime-closed-loop`、`make delivery-pipeline` 和 `release-image` workflow，而不是新增另一套首发专用发布机制。

备选方案：

- 新建一个完全独立的首发发布脚本或工作流。

不采用的原因：

- 会引入第二条发布路径，增加维护成本和语义分叉，不利于后续持续发布。

### 决策 4: StatefulSet 自动写动作不作为首发阻断项

首发发布执行必须明确把 StatefulSet 自动写动作排除在首发验收之外；StatefulSet reality 验证继续保留为独立、显式启用的后续验证入口。

备选方案：

- 把 StatefulSet reality 通过作为首发 go-live 的前置条件。

不采用的原因：

- 这会扩大首发范围，延迟第一个可交付版本，也和当前 README 对首发范围的声明不一致。

### 决策 5: 首发证据包必须由决策包、演练输出和回滚说明三部分组成

首发发布记录至少包含：go-live decision pack、pilot/cutover decision pack、预生产演练输出、版本标签记录、回滚说明与当前版本边界说明。

备选方案：

- 仅依赖 CI 绿灯或零散文档说明作为首发发布依据。

不采用的原因：

- 无法满足“可追踪、可回滚、可复盘”的首发交付要求。

## 风险 / 权衡

- [风险] 发布执行变更可能与现有 go-live、pilot、镜像发布规格出现重复描述
  → 缓解措施：新增 capability 只负责“首发执行闭环”，其余能力继续在原有 capability 中增量补充。
- [风险] 团队把 RC 与稳定版步骤视为形式化流程，实际仍手工跳过
  → 缓解措施：要求发布文档、decision pack 和镜像标签顺序保持一致，缺任一证据即不得宣称首发完成。
- [风险] 首发范围如果写得不够收敛，后续实现会继续膨胀
  → 缓解措施：在规格中显式声明 StatefulSet 自动写动作和额外 UI 不属于首发阻断范围。
- [权衡] 复用现有脚本意味着首发流程对当前脚本命名和输出格式有依赖
  → 缓解措施：将这些输出字段写入规格与任务，后续如调整脚本也必须保持契约稳定。

## Migration Plan

1. 新增 `v1-release-execution` capability，并为 go-live、pilot/cutover、镜像分发、本地联调添加首发执行相关增量需求。
2. 更新发布与回滚文档，形成单一路径：预生产演练、RC 发布、pilot/cutover、稳定版发布、回滚基线。
3. 校验 `release-image` workflow、delivery pipeline 与演练入口能够支撑该路径，并补齐缺失的发布记录说明。
4. 基于该流程执行一次真实首发预演，生成首发证据包。
5. 若预演通过，再按同一流程发布首个稳定版；若失败，则按回滚说明退回 RC 或继续停留在预生产阶段。

## Open Questions

- 首发 pilot 观察窗口是以固定时长定义，还是允许按环境风险等级调整？
- 首发证据包最终落在仓库文档、release assets，还是外部运维归档系统中？
