## ADDED Requirements

### 需求:首发预生产演练必须复用现有联调入口

系统必须要求首发版本的预生产演练复用现有最小安装、本地开发回路与最小 smoke 入口，禁止为首发发布单独引入一套与日常验证脱节的演练路径。

#### 场景: 执行首发预生产演练

- **当** 团队为首发版本执行预生产验证
- **那么** 系统必须能够通过 install、dev-loop、smoke 与 delivery pipeline 的既有入口完成验证，并输出可归档的演练证据

#### 场景: 首发演练尝试绕过统一入口

- **当** 首发发布计划试图用临时手工步骤替代既有 install、dev-loop、smoke 验证入口
- **那么** 系统必须要求补充统一入口下的验证结果，禁止仅凭临时步骤宣称首发验证完成

### 需求:首发演练记录必须区分发布验证与后续扩展验证

系统必须在首发预生产演练记录中区分“Deployment 首发发布验证”和“StatefulSet reality 等后续扩展验证”，禁止把二者混写为同一首发完成证据。

#### 场景: 生成首发预生产演练记录

- **当** 系统记录一次首发预生产演练输出
- **那么** 演练记录必须明确说明哪些验证属于首发阻断范围，哪些仅属于后续扩展能力验证

#### 场景: 演练记录混淆首发与扩展验证

- **当** 预生产演练把 StatefulSet reality 或其他扩展验证混写为首发必需结果
- **那么** 系统必须要求更正记录，以免扩大首发范围
