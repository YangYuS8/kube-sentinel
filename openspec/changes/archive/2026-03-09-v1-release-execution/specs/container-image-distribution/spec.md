## ADDED Requirements

### 需求:首发镜像发布必须区分 RC 与稳定版推进

系统必须在首发发布流程中先发布预发布标签（如 `vX.Y.Z-rc.N`）用于预生产与 pilot/cutover 验证，并且只有在首发放行后才允许发布稳定版标签。

#### 场景: 发布首发 RC 镜像

- **当** 团队开始首发版本验证
- **那么** 系统必须允许先发布 RC 镜像并将其作为预生产与 pilot/cutover 的目标版本

#### 场景: 首发验证通过后发布稳定版

- **当** RC 版本完成首发所需验证并获得放行结论
- **那么** 系统必须允许再发布稳定版标签，并将其作为首个正式可交付版本

#### 场景: 首发验证未完成试图发布稳定版

- **当** 操作者在未完成 RC 验证或未获得放行结论前推送稳定版标签
- **那么** 系统必须将该行为视为不符合首发发布流程

### 需求:latest 只能在首发稳定版发布时更新

系统必须确保首发阶段的 `latest` 仅在稳定版标签发布时更新，禁止任何 RC 或手动补发流程把 `latest` 指向未完成首发验证的版本。

#### 场景: 首发稳定版更新 latest

- **当** 团队发布首个稳定版标签
- **那么** 系统必须允许同时更新 `latest`

#### 场景: RC 或手动补发尝试覆盖 latest

- **当** 首发阶段的 RC 或手动补发流程试图更新 `latest`
- **那么** 系统必须阻止其覆盖稳定版入口
