## 新增需求

### 需求:值班模式部署必须具备快速启用与快速退出能力
系统必须为值班模式提供快速启用与快速退出路径，至少支持最小安装、最小验证、关闭自动动作和退回只读模式，禁止让使用者只能在完整平台式安装后才能试用 V1。

#### 场景: 团队试点启用值班模式
- **当** 团队在测试环境或低风险环境中试点启用 Kube-Sentinel V1
- **那么** 系统必须提供最小安装、验证和关闭自动动作的清晰入口

## 修改需求

### 需求:本地与测试环境必须具备最小可安装入口
系统必须为 Kube-Sentinel 提供一个最小可安装入口，使测试环境能够以统一方式完成 CRD 安装、控制器部署与默认保守配置加载，禁止仅依赖分散的手工步骤作为首选部署路径。

#### 场景: 新环境执行最小安装
- **当** 开发者或测试人员在一个新的 Kubernetes 测试环境中安装 Kube-Sentinel
- **那么** 系统必须能通过统一入口完成 CRD、控制器与默认保守配置的安装，并输出可继续执行 smoke 验证的下一步说明

### 需求:本地开发回路必须支持最小 smoke 验证
系统必须提供面向本地调试的最小 smoke 验证入口，至少覆盖 Deployment 告警接入、HealingRequest 创建、默认安全阻断路径和单次 L1 成功路径。

#### 场景: 执行本地 smoke 验证
- **当** 开发者在满足前置条件的本地环境执行 smoke 验证入口
- **那么** 系统必须能够验证默认阻断语义与单次 L1 成功闭环，并输出对应的对象状态与结果说明

## 移除需求

### 需求:首发预生产演练必须复用现有联调入口
**Reason**: 该需求把值班工具的本地采用路径与首发发布治理绑定过紧，不适合作为极简 V1 的核心采用要求。
**Migration**: 如仍需要预生产发布演练，可在后续发布治理变更中复用 install/dev-loop/smoke 入口。

### 需求:首发演练记录必须区分发布验证与后续扩展验证
**Reason**: 该需求服务发布管理和证据归档，不属于 V1 工具化部署体验。
**Migration**: 迁移到独立的发布治理或执行流程变更。
