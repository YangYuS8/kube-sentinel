## 新增需求

### 需求:Agent 必须提供值班事件摘要
系统必须为每个进入值班处理路径的 incident 提供结构化事件摘要，至少包括目标 workload、触发来源、当前阶段、最近动作、阻断或失败原因和关联键，禁止仅输出无法追踪的自由文本总结。

#### 场景: 生成单个 incident 摘要
- **当** 运维通过 Agent 查询某个 incident 的当前情况
- **那么** Agent 必须返回包含 workload、trigger、runtime decision、reason 和 correlationKey 的结构化摘要

### 需求:Agent 必须提供下一步建议
系统必须允许 Agent 基于 runtime 状态、事件和基础指标生成下一步建议，且建议必须区分“建议查看”“建议人工执行”和“暂不建议执行”的动作类别。

#### 场景: incident 被阻断时生成建议
- **当** incident 因门禁、证据不足或动作失败进入阻断或只读路径
- **那么** Agent 必须提供清晰的下一步建议，并说明当前不建议继续自动动作的原因

### 需求:Agent 必须支持值班交接说明
系统必须允许 Agent 为单个 incident 生成简明交接说明，至少包括触发时间、目标对象、系统已执行动作、当前结论和建议接手动作。

#### 场景: 夜间转交白天值班
- **当** 运维需要将未完全解决的 incident 交接给下一位处理者
- **那么** Agent 必须生成可复制的交接说明，并保留稳定关联键以便继续排查

### 需求:Agent 必须遵守只读边界
Agent 在 V1 中必须默认保持只读，禁止将自由规划结果直接作为生产写动作执行输入。

#### 场景: Agent 遇到高风险修复建议
- **当** Agent 推断某个 incident 可能需要高风险、不可逆或超出默认动作集合的处理
- **那么** Agent 必须将其标记为人工建议，而不得直接触发生产写路径
