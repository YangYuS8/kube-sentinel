## 新增需求

### 需求:基础观察面必须服务值班理解与人工接管
系统必须将基础观察面限定为帮助值班人员快速理解 incident 与完成人工接管，至少覆盖触发情况、最小动作结果、阻断原因、快照结果和稳定关联键，禁止将发布治理趋势视为 V1 基础观察面的前置内容。

#### 场景: 值班人员查看单个 incident 的观察证据
- **当** 值班人员查看某个 incident 的运行证据
- **那么** 系统必须提供与该 incident 相关的基础事件、审计记录、关键指标和关联键

## 修改需求

### 需求:控制台指标必须具备标准抓取入口
系统必须以标准 Prometheus 抓取方式暴露控制器基础指标，使 Grafana 能通过 Prometheus 数据源消费 incident 触发、动作结果、阻断原因和快照结果等最小运行态指标，禁止仅依赖临时端口转发或人工拼接查询规则作为唯一接入方式。

#### 场景: 测试环境接入 Grafana
- **当** 测试环境启用 Grafana 面板查看 Kube-Sentinel 运行态指标
- **那么** 系统必须提供标准的指标抓取入口与最小接入配置，使 Prometheus 能稳定抓取 incident 相关的基础控制器指标

### 需求:指标视图必须覆盖最小运维分组
系统必须为 Grafana 等指标视图定义最小运维分组，至少覆盖总体触发与成功率、L1 动作结果、快照结果和阻断趋势，禁止将发布治理或复杂交付趋势作为 V1 最小仪表盘的必需内容。

#### 场景: 运维查看最小仪表盘
- **当** 运维打开 Kube-Sentinel 的最小运行态仪表盘
- **那么** 仪表盘必须能够按分组展示总体触发、L1 结果、快照结果与阻断趋势四类视图

## 移除需求

### 需求:发布就绪度指标暴露
**Reason**: 发布就绪度属于发布治理能力，不属于 V1 值班核心观察面。
**Migration**: 如后续仍需保留，应在独立的发布治理变更中维护。

### 需求:值班告警映射最小集
**Reason**: 现有内容主要围绕发布阻断、审批缺失和演练证据过期，不符合极简 V1 观察面定位。
**Migration**: 将相关规则迁移到发布治理或 operator workflow 资产。

### 需求:交付运营指标暴露
**Reason**: 该需求主要服务持续交付治理，不属于 V1 运行态最小观察面。
**Migration**: 保留为未来的交付治理能力。

### 需求:周期汇总报告输出
**Reason**: 周期汇总报告面向治理和汇报，不是夜间值班的首发必需能力。
**Migration**: 如有需要，后续作为外围分析能力重新引入。
