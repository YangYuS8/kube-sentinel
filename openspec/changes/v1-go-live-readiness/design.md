## 上下文

当前项目在检测、自愈、门禁、演练和审计维度已具备较高覆盖，但“是否允许上线”的最终判定仍分散在多个脚本与文档中。首个可上线版本需要把关键证据统一收敛为可审计的 go/no-go 决策链路，确保值班与发布角色在固定窗口内快速、稳定地做出一致决策。

## 目标 / 非目标

**目标：**
- 建立 V1 go-live 闸门的最小通过条件与阻断条件。
- 建立 release decision pack 的标准字段与生成流程。
- 建立“预生产验证通过 -> 生产灰度放量”的单向流程与失败回退策略。
- 将冻结窗口、人工覆盖、审批审计收敛为一致规则。

**非目标：**
- 不新增新的自愈策略层级与业务动作类型。
- 不引入新的外部系统依赖。
- 不扩大 CRD 功能面，仅补齐上线决策语义与流程约束。

## 决策

1. 采用“统一决策包 + 固定闸门规则”作为上线入口，避免按角色各自解读指标。
2. 将闸门分为五类：质量门禁、运行稳定性、演练与回滚、审批与冻结、审计完整性；任一失败即 block。
3. 预生产验证通过是生产灰度的前置条件，禁止跨阶段直接放量。
4. 人工覆盖仅允许在冻结窗口外、审批齐全且审计字段完整时触发，并且必须保留关联键。
5. 所有 go/no-go 决策必须生成结构化证据包，作为发布后复盘基线。

## 风险 / 权衡

- [风险] 规则过严可能降低发布频率  
  → 缓解：定义 degrade 保护路径与时限内修复窗口。
- [风险] 决策包字段过多导致使用负担  
  → 缓解：严格控制最小字段集，避免冗余指标。
- [权衡] 增加上线前准备成本，换取可追溯性与生产稳定性。

## 迁移计划

1. 在新 capability 中定义 V1 go-live 闸门与决策包规范。
2. 在 `delivery-quality-gates` 增量规范中加入 go-live 闸门与证据输出约束。
3. 在 `release-readiness-and-oncall-automation` 中补齐值班审批与覆盖审计一致性约束。
4. 在 `runtime-closed-loop-validation` 中补齐预生产到生产放量门禁语义。
5. 实施后进行一次预生产全链路演练，确认闸门可执行。

## 开放问题

- 是否需要对不同环境（staging/preprod/prod）采用不同阈值模板？
- 决策包是否需要按发布批次自动聚合历史窗口趋势？
- 对紧急漏洞修复场景，例外流程是否需要单独阈值策略？
