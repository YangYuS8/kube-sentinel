## 为什么

当前能力覆盖已较完整，但缺少面向“首个可上线版本”的统一闸门收敛标准，导致发布决策依赖人工拼接信息，影响上线速度与一致性。需要一个面向 V1 的最小上线决策框架，把质量门禁、演练证据、回滚可用性和值班审批收敛为可执行的 go/no-go 判定。

## 变更内容

- 定义 V1 上线闸门（go-live gate）最小通过条件，覆盖质量门禁、SLO、演练、回滚、审批。
- 定义发布决策包（release decision pack）最小字段和生成时机，支持值班快速判定。
- 定义回滚演练达标阈值与失败处理流程，确保“可回滚优先”落地。
- 定义冻结窗口与人工覆盖规则，明确何时允许例外放行及审计要求。
- 明确预生产验证到生产灰度放量的最小流程与阻断条件。

## 功能 (Capabilities)

### 新增功能

- `v1-go-live-readiness`: 提供 V1 上线决策闸门、证据包和放量流程的统一规范。

### 修改功能

- `delivery-quality-gates`: 增加 V1 go-live 闸门判定与 release decision pack 输出要求。
- `release-readiness-and-oncall-automation`: 增加值班审批、冻结窗口、人工覆盖与上线决策一致性要求。
- `runtime-closed-loop-validation`: 增加“预生产验证通过才允许生产灰度”的验收语义与失败阻断路径。

## 影响

- 主要影响 OpenSpec 发布流程与门禁语义，不直接扩展核心自愈功能边界。
- 影响 CI/发布操作手册与值班决策流程，提升上线判定一致性。
- 不引入新的业务 API 面向用户的破坏性变更。
