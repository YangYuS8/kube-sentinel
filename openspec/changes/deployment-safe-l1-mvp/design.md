## 上下文

Kube-Sentinel 当前已经具备告警接入、HealingRequest 协调、编排、门禁、快照和审计的基本骨架，但行为范围同时覆盖多工作负载、多级处置和多类治理能力。对首个可持续交付版本而言，真正的问题不是“能力是否存在”，而是“边界是否足够收敛、是否能被稳定验证”。

本设计将首发范围压缩为一个真实的纵向切片：仅允许 Deployment 进入可写的 L1 自动动作路径，所有其他更激进或更复杂的能力继续保留在只读评估、阻断或后续变更中。这样可以让当前代码路径围绕一条可证明安全的主流程达成一致，并把验收基线落在最关键的治理要求上：幂等、写前快照、安全门禁、失败阻断、状态可解释和证据可检索。

## 目标 / 非目标

**目标：**

- 形成从 Alertmanager Webhook 到 Deployment L1 自动动作的端到端安全闭环。
- 将维护窗口、速率限制、爆炸半径、熔断和写前快照统一视为 L1 写路径的前置条件。
- 定义首发版本的稳定状态语义和审计证据，确保每次处理都有明确阶段、动作结果、失败原因和下一步建议。
- 为后续 L2/L3、StatefulSet、恢复编排和更复杂门禁保留兼容的扩展位置，而不在首发版本提前承诺。

**非目标：**

- 不在本变更中交付 Deployment L2 自动回滚或 L3 自动升级编排。
- 不在本变更中开放 StatefulSet 条件可写或任何默认可写路径。
- 不在本变更中交付快照恢复执行、发布演练自动化或交付运营报表。
- 不在本变更中引入新的外部依赖、独立存储组件或旁路执行器。

## 决策

### 决策: 将首发写路径限定为 Deployment L1 单动作

首发版本只允许 Deployment 执行单一、最小影响的 L1 动作，并把其余阶段视为后续增量。这样可以避免在第一个交付物中同时承担多阶段状态机、候选版本判定和恢复编排的复杂性。

备选方案：

- 直接交付 L1-L2-L3 全链路。未采用，因为当前验收面过大，任何一段证据不足都会拖慢首个成品落地。
- 只做只读评估，不执行任何写动作。未采用，因为虽然更保守，但不足以验证“安全自动处置”这条产品主价值链。

### 决策: 所有 L1 写动作都必须通过“门禁 + 快照”双前置

是否可以执行 L1，不由告警本身决定，而由一组前置约束共同裁决：维护窗口、速率限制、爆炸半径、熔断和持久快照创建。任何一个环节失败，都必须终止写操作并输出阻断证据。

备选方案：

- 允许部分门禁失败时继续执行。未采用，因为与“无法证明安全时默认降级人工介入”的核心原则冲突。
- 先执行 L1，再异步补快照。未采用，因为这会破坏“可回滚优先”的前提。

### 决策: 用状态语义而不是自由文本表达首发结果

首发版本需要把每次处理结果稳定映射到固定状态字段，例如阶段、最近动作、阻断原因、门禁裁决和下一步建议，而不是依赖零散日志或难以检索的错误文本。这样做的目的不是美化状态，而是为测试、演练和后续自动化判定提供稳定接口。

备选方案：

- 只输出日志和事件。未采用，因为它们对人类可读，但对契约验证不稳定。
- 为每种失败设计独立 CRD。未采用，因为会让首发范围和迁移成本显著扩大。

### 决策: 通过 Capability 增量规范来冻结首发范围

本变更不新造一套平行规范，而是在现有接入、编排、快照、熔断和 API 契约能力上追加首发 MVP 要求，再增加一个专门描述首发范围的新 capability。这样既能复用既有规格体系，又能避免首发边界被后续长期能力模糊化。

备选方案：

- 将所有首发要求塞进一个新的总规范。未采用，因为会割裂现有 capability 归属，后续归档时难以维护。

## 风险 / 权衡

- [风险] 首发版本只支持 Deployment L1，演示面会显得保守。 -> 缓解措施：在 proposal 和 tasks 中明确这是“首个可持续交付基线”，而不是最终能力范围。
- [风险] 当前代码已存在部分 L2/L3 路径，若不显式冻结，后续实现时容易继续扩范围。 -> 缓解措施：在 specs 和 tasks 中把 L2/L3 标注为非目标，并将验收只绑定到 L1 闭环。
- [风险] 快照、门禁和状态语义涉及多个模块，首发仍然是跨模块改动。 -> 缓解措施：以端到端场景组织任务，避免按模块分散交付，优先验证写前阻断和成功路径。
- [风险] 过早承诺过多可观测指标会拖慢交付。 -> 缓解措施：首发只要求最小可诊断集，复杂趋势分析留给后续变更。

## 迁移计划

- 先完成 proposal、design、specs、tasks，冻结首发范围与非目标。
- 实现阶段优先补齐 Deployment L1 成功路径与失败阻断路径的测试，再补 CRD/状态语义与审计输出。
- 交付前执行单元测试、关键并发路径 race 测试、vet、lint、CRD 一致性校验和质量门禁脚本。
- 如首发行为在验证中不满足安全前置条件，回退到只读评估模式，不允许以“先上线再修复”方式放行。

## 开放问题

- 首发 L1 动作是否严格限定为 rollout restart，还是允许在保持“最小影响”前提下支持少量可配置动作。
- 首发阶段的最小指标集是否单独作为 observability capability 增量记录，还是暂时由编排与审计要求隐含覆盖。
- 对已有部分 L2/L3 代码路径，首发实现是显式禁用还是仅通过门禁与任务范围不触达。
