# statefulset-reality-validation-and-hardening Design

## 上下文

当前 Kube-Sentinel 已经具备 StatefulSet 受控自动动作、L2 健康版本回滚、冻结窗口与 snapshot restore 的主要领域语义，相关状态字段、指标和单元测试也已存在。但与 Deployment 刚完成的 minikube 真实性验证相比，StatefulSet 仍主要停留在“规格与 fake 测试成立”的阶段，尚未证明真实 Kubernetes 中的 ControllerRevision 历史、分区滚动语义和失败恢复联动是否与当前假设一致。

这带来两个直接风险：

- 现有适配器可能把 fake client 中成立的候选选择或恢复假设，错误地投射到真实集群。
- 交付门禁虽然要求可回滚与失败证据完整，但目前对 StatefulSet 还缺少可重复执行的真实性验证入口。

这次变更应当延续刚完成的 Deployment 路径经验：先通过真实集群验证找出差距，再以最小增量修正适配器、状态语义或验证脚本，并把真实性检查固化成仓库测试资产。

## 目标 / 非目标

**目标：**

- 为 StatefulSet L1 受控动作、L2 候选选择、L2 回滚、冻结与 snapshot restore 建立真实集群验证闭环。
- 把真实性验证沉淀为可重复执行的集成测试或受控演练入口，避免问题只在一次性调试中暴露。
- 当真实集群行为与当前实现假设不一致时，只做最小必要修正，并保持安全优先与失败降级原则。
- 对齐运行时闭环验证与发布文档，使 StatefulSet 真实性验证成为交付前可声明的证据。

**非目标：**

- 不在这次变更中扩展新的 StatefulSet 自愈级别或引入更激进的自动化动作。
- 不重写整套 StatefulSet 编排架构，仅修正被真实集群证伪的具体假设。
- 不把生产 cutover 自动化或快照治理体系一并重做；仅在 StatefulSet 真实性验证需要时触及相关文档或验证入口。
- 不要求默认 CI 在所有环境下运行真实集群集成测试；真实集群检查应继续保持显式启用。

## 决策

### 决策 1：先验证真实 StatefulSet 语义，再决定是否修代码

- 选择：优先新增或扩展受环境变量控制的真实集群集成测试，覆盖 StatefulSet 的 revision 历史、候选筛选、回滚与冻结联动，再根据验证结果决定是否修正适配器或编排器。
- 原因：当前最大不确定性不在抽象设计，而在真实 Kubernetes 语义。先证明问题，才能避免为不存在的问题扩展实现。
- 备选方案：
  - 直接按假设修改适配器：风险是重复 Deployment 之前的错误路径，先写出“看起来对”的逻辑，再被真实集群推翻。
  - 仅补单元测试：无法验证 ControllerRevision、分区滚动和真实恢复链路。

### 决策 2：集成测试保持显式启用，不纳入默认快速测试路径

- 选择：真实性验证使用显式环境变量或明确入口启用，默认 `go test ./...` 仍以快速、稳定为目标。
- 原因：真实集群测试依赖 kube context、镜像、节点状态和资源准备，强行纳入默认测试会把局部实现风险升级为开发流程噪音。
- 备选方案：
  - 默认执行所有真实性测试：在缺少集群或资源不足时会制造大量伪失败。
  - 完全依赖手工演练脚本：可重复性差，回归信号弱。

### 决策 3：StatefulSet 修正遵循“最小假设修复”原则

- 选择：如果真实集群表明当前候选选择、冻结或恢复语义存在问题，则仅修正具体不成立的判断条件、状态迁移或证据输出，不借机重构整个编排器。
- 原因：当前已有大量状态字段、指标和测试基线。大范围重构会把真实性修复和架构重组混在一起，增加回归面。
- 备选方案：
  - 顺手统一 Deployment 与 StatefulSet 的全部 L2 流程：长期可能有价值，但不适合作为真实性修复的首轮动作。

### 决策 4：规格层同时补“真实性验证要求”，而不仅是运行时行为

- 选择：除了细化 StatefulSet 运行时能力，还在 `runtime-closed-loop-validation` 中增加或修改要求，明确 StatefulSet 必须有真实集群闭环验证入口与阻断条件。
- 原因：如果只修实现，不把验证本身纳入规范，未来很容易再次退回到“单测成立但真实集群未证实”的状态。
- 备选方案：
  - 只修改 StatefulSet 相关规范：会遗漏交付前验证这一层契约。

## 风险 / 权衡

- [风险] StatefulSet 真实集群测试比 Deployment 更脆弱，尤其受存储、分区滚动与 ControllerRevision 历史影响。
  -> 缓解：控制测试场景为最小可复现路径，只验证最关键的语义断言。
- [风险] 不同 Kubernetes 版本对 ControllerRevision 或 rollout 可见性的细节可能不同。
  -> 缓解：优先依赖稳定元数据与最终状态，而不是脆弱的瞬时字段。
- [风险] 真实性修复可能诱发已有单元测试基线调整。
  -> 缓解：先用真实集群证据确定根因，再最小化修改 fake 测试断言。
- [风险] 把真实集群验证写入规格后，团队可能误解为必须在所有 CI 环境执行。
  -> 缓解：在设计和任务中明确“显式启用”与“默认跳过”的边界。

## 迁移计划

1. 先补齐提案对应的增量规范，明确 StatefulSet 真实性验证与闭环验证要求。
2. 新增受显式环境变量控制的真实集群验证入口，优先覆盖最小关键路径。
3. 在真实集群执行验证，记录与当前实现假设不一致的具体点。
4. 仅对被证伪的判断条件或状态迁移进行修正，并补充对应单元测试。
5. 更新发布/回滚与闭环验证文档，把 StatefulSet 真实性检查纳入验收说明。

## 开放问题

- StatefulSet 真实性验证应优先基于 minikube 还是更接近生产的多节点测试环境？
- 是否需要把 StatefulSet 真实性测试入口并入现有 `drill-runtime-closed-loop.sh`，还是保持独立入口更清晰？
- 如果真实集群显示 ControllerRevision 选择策略不稳定，是否应补充显式稳定窗口或额外版本证据，而不只依赖当前适配器查询？
